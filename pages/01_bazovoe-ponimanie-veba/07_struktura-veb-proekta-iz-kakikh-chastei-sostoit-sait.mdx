# Структура веб-проекта: из каких частей обычно состоит сайт

## Коротко

Веб-проект — это не один файл и не одна страница. Даже небольшой сайт обычно состоит из нескольких частей: интерфейса, файлов, логики, данных, настроек и инструментов для запуска.

В простом проекте все это может находиться в нескольких папках. В более сложном веб-приложении frontend, backend, база данных и инфраструктура могут быть разделены на самостоятельные системы.

Понимание структуры проекта помогает:

* быстрее ориентироваться в коде;
* правильно объяснять задачу ИИ-ассистенту;
* понимать, какой файл нужно изменить;
* не смешивать интерфейс, данные и серверную логику;
* проверять, не пропустил ли ИИ важную часть задачи.

Главный вопрос при знакомстве с проектом звучит так:

> Из каких частей он состоит и за что отвечает каждая из них?

## Что такое структура веб-проекта

Структура веб-проекта — это способ организации его частей, файлов и связей между ними.

Она показывает:

* где находится интерфейс;
* где хранится серверная логика;
* где описаны страницы;
* где находятся стили;
* как подключаются данные;
* где лежат изображения;
* какие настройки нужны для запуска;
* как проект собирается и публикуется.

Структура помогает разработчику и ИИ-ассистенту понимать назначение каждого элемента.

Без понятной организации проект быстро превращается в набор файлов, в котором сложно определить:

* где искать нужную функцию;
* что можно изменить безопасно;
* какие части зависят друг от друга;
* почему возникла ошибка;
* как добавить новую возможность.

## Все ли сайты устроены одинаково

Нет.

Структура зависит от:

* размера проекта;
* используемых технологий;
* наличия backend;
* способа хранения данных;
* количества страниц;
* состава команды;
* требований к публикации;
* выбранной архитектуры.

Простой лендинг может состоять из HTML, CSS, JavaScript и изображений.

Веб-приложение может включать:

* frontend;
* backend;
* API;
* базу данных;
* авторизацию;
* файловое хранилище;
* административную панель;
* интеграции;
* тесты;
* инструменты сборки;
* конфигурацию публикации.

Поэтому не существует одной обязательной структуры для всех веб-проектов.

Но есть типовые части, которые встречаются очень часто.

## Основные части веб-проекта

На базовом уровне веб-проект можно разделить на несколько крупных областей:

1. Frontend.
2. Backend.
3. База данных.
4. Статические файлы.
5. Конфигурация.
6. Зависимости.
7. Инструменты разработки.
8. Тесты и документация.
9. Инфраструктура публикации.

Не каждый проект содержит все эти части.

Например, статический сайт может не иметь собственного backend и базы данных.

## Frontend

Frontend — это клиентская часть проекта, которая работает в браузере.

Она отвечает за:

* страницы;
* кнопки;
* формы;
* меню;
* тексты;
* изображения;
* отображение данных;
* реакцию на действия пользователя;
* отправку запросов на backend.

В простом проекте frontend может состоять из файлов:

* HTML;
* CSS;
* JavaScript.

В современном проекте frontend часто собирается из компонентов.

Компонент — это отдельная часть интерфейса, которую можно использовать и изменять независимо.

Например:

* кнопка;
* карточка товара;
* поле формы;
* меню;
* модальное окно;
* таблица;
* календарь.

Компонентный подход помогает не хранить весь интерфейс в одном огромном файле.

## Страницы и маршруты frontend

Если сайт содержит несколько разделов, в проекте обычно есть страницы или маршруты.

Например:

* главная;
* услуги;
* врачи;
* контакты;
* личный кабинет;
* настройки;
* административная панель.

Маршрут определяет, какой интерфейс показывается по определенному адресу.

Например:

* `/` — главная страница;
* `/services` — список услуг;
* `/doctors` — список врачей;
* `/profile` — профиль пользователя;
* `/admin` — административная панель.

В зависимости от технологии страницы могут быть отдельными файлами или собираться из компонентов.

## Стили

Стили отвечают за внешний вид интерфейса.

Они определяют:

* цвета;
* шрифты;
* размеры;
* отступы;
* расположение блоков;
* адаптацию под разные экраны;
* состояния кнопок;
* анимации.

Стили могут храниться:

* в одном общем CSS-файле;
* в отдельных файлах компонентов;
* в CSS-модулях;
* в специальных библиотеках;
* прямо рядом с кодом компонента.

Для начинающего важнее не конкретный способ, а понимание того, что структура и оформление интерфейса обычно разделены.

## JavaScript и логика интерфейса

JavaScript отвечает за поведение страницы.

Например:

* открывает меню;
* переключает вкладки;
* проверяет форму;
* отправляет запрос;
* показывает сообщение;
* обновляет данные без перезагрузки;
* управляет состоянием интерфейса.

В большом проекте эту логику разделяют по назначению.

Отдельно могут находиться:

* компоненты;
* функции для запросов;
* обработчики форм;
* утилиты;
* состояние приложения;
* маршруты;
* настройки.

Это помогает избежать ситуации, когда один файл одновременно отображает интерфейс, запрашивает данные, проверяет права и содержит десятки несвязанных функций.

## Backend

Backend — это серверная часть проекта.

Она принимает запросы от frontend или других клиентов и выполняет серверные операции.

Backend может отвечать за:

* регистрацию;
* вход;
* проверку прав;
* работу с базой данных;
* создание заказов;
* запись на прием;
* расчет стоимости;
* отправку писем;
* интеграции;
* формирование отчетов;
* обработку файлов.

В структуре backend часто можно встретить:

* маршруты;
* контроллеры;
* сервисы;
* модели данных;
* middleware;
* настройки;
* подключение к базе;
* обработку ошибок.

Названия могут отличаться в зависимости от языка и фреймворка.

## Маршруты backend

Маршруты backend определяют, какие запросы принимает сервер.

Например:

* `GET /api/doctors` — получить список врачей;
* `POST /api/appointments` — создать запись;
* `PATCH /api/profile` — изменить профиль;
* `DELETE /api/appointments/123` — отменить запись.

Маршрут связывает адрес и HTTP-метод с определенной серверной функцией.

Обычно сам маршрут не должен содержать всю бизнес-логику. Он передает запрос следующему слою, который выполняет нужную операцию.

## Контроллеры и обработчики

Контроллер или обработчик получает запрос и управляет его обработкой.

Он может:

1. Получить данные из запроса.
2. Вызвать проверку.
3. Передать данные сервису.
4. Получить результат.
5. Вернуть HTTP-ответ.

В небольшом проекте контроллер может быть простой функцией.

В большом проекте его стараются не перегружать логикой, чтобы код оставался понятным.

## Сервисы и бизнес-логика

Сервис — это часть backend, в которой часто размещается бизнес-логика.

Бизнес-логика — это правила работы конкретного проекта.

Например:

* запись можно создать только на свободное время;
* скидка доступна определенной категории клиентов;
* отмена разрешена не позднее установленного срока;
* менеджер видит только свой филиал;
* администратор может изменять расписание.

Такие правила не относятся напрямую ни к интерфейсу, ни к базе данных. Они описывают, как должна работать система.

Поэтому их полезно хранить отдельно от маршрутов и технических настроек.

## База данных

База данных хранит информацию проекта.

Например:

* пользователей;
* товары;
* заказы;
* записи;
* врачей;
* услуги;
* сообщения;
* настройки;
* платежи;
* статусы.

Backend обращается к базе данных, чтобы:

* получить информацию;
* создать новые записи;
* изменить существующие;
* удалить данные;
* проверить связи и ограничения.

Frontend обычно не должен иметь прямой полный доступ к базе.

## Модели и схемы данных

Модель или схема описывает, какие данные хранит объект.

Например, запись на прием может содержать:

* идентификатор;
* пациента;
* врача;
* дату;
* время;
* статус;
* комментарий;
* дату создания.

Схема помогает определить:

* какие поля обязательны;
* какие типы данных допустимы;
* как объекты связаны;
* какие ограничения действуют.

При работе с ИИ важно сообщать структуру данных, иначе он может придумать несовместимые поля или неверные связи.

## Миграции

Миграции описывают изменения структуры базы данных.

Например:

* добавить поле;
* создать таблицу;
* изменить тип данных;
* добавить связь;
* удалить устаревший столбец.

Они нужны, чтобы изменения базы можно было последовательно применять на разных средах: у разработчика, на тестовом сервере и в опубликованном проекте.

Изменить только код модели недостаточно, если реальная база данных осталась прежней.

## Статические файлы

Статические файлы — это ресурсы, которые сервер или хостинг обычно отдает без дополнительной обработки.

К ним относятся:

* изображения;
* иконки;
* шрифты;
* видео;
* PDF-документы;
* готовые CSS- и JavaScript-файлы;
* файлы для скачивания.

Для них часто создают папки с названиями вроде:

* `public`;
* `assets`;
* `static`;
* `images`.

Название зависит от технологии.

Важно понимать различие между файлами проекта и пользовательскими файлами.

Например, логотип является частью проекта, а фотография, загруженная пользователем, обычно хранится отдельно — в облачном хранилище или специальной файловой системе.

## Конфигурационные файлы

Конфигурация определяет, как проект запускается, собирается и взаимодействует с окружением.

Она может содержать:

* настройки сборки;
* адреса сервисов;
* параметры сервера;
* правила форматирования;
* маршруты;
* настройки базы данных;
* команды запуска;
* параметры публикации.

Примеры конфигурационных файлов зависят от технологии.

Новичку важно понимать: файл конфигурации часто не содержит основную функцию сайта, но без него проект может не запускаться.

## Переменные окружения

Переменные окружения хранят настройки, которые могут отличаться в разных средах.

Например:

* адрес базы данных;
* URL backend;
* секретный ключ;
* режим запуска;
* данные внешнего сервиса;
* почтовые настройки.

Часто для локальной разработки используется файл `.env`.

Секретные значения нельзя публиковать в открытом репозитории или помещать во frontend.

В опубликованном проекте переменные окружения обычно задаются в настройках хостинга.

## Зависимости

Современный проект редко пишется полностью с нуля.

Он использует библиотеки и пакеты для:

* интерфейса;
* маршрутизации;
* форм;
* работы с датами;
* подключения к базе;
* авторизации;
* тестирования;
* сборки.

Список зависимостей хранится в специальном файле проекта.

Например, в JavaScript-проектах часто используется `package.json`.

Этот файл может содержать:

* название проекта;
* команды запуска;
* список библиотек;
* версии зависимостей;
* служебные настройки.

Папку с установленными библиотеками обычно не редактируют вручную.

## Инструменты сборки

Исходный код не всегда отправляется пользователю в том виде, в котором его написал разработчик.

Перед публикацией проект может проходить сборку.

Во время сборки:

* объединяются файлы;
* преобразуется код;
* оптимизируются ресурсы;
* уменьшается размер файлов;
* проверяются ошибки;
* создается готовая версия для публикации.

Инструменты сборки особенно часто используются в современных frontend-проектах.

Поэтому важно различать:

* исходный код;
* собранную версию;
* опубликованные файлы.

## Тесты

Тесты проверяют, правильно ли работает проект.

Они могут проверять:

* отдельную функцию;
* компонент;
* API;
* бизнес-правило;
* пользовательский сценарий;
* взаимодействие нескольких частей.

Например, тест может проверять, что сервер не позволяет создать запись на уже занятое время.

Тесты часто находятся в отдельных папках или рядом с проверяемым кодом.

Даже если новичок не пишет тесты самостоятельно, полезно понимать их роль и просить ИИ обновлять тесты вместе с кодом.

## Документация

Документация объясняет устройство и использование проекта.

К ней могут относиться:

* README;
* инструкции по запуску;
* описание API;
* правила работы с кодом;
* схема архитектуры;
* описание переменных окружения;
* порядок публикации.

README — это обычно первый файл, который стоит открыть при знакомстве с проектом.

В хорошем README указано:

* что делает проект;
* какие технологии используются;
* как его установить;
* как запустить;
* какие настройки нужны;
* как выполнить сборку;
* как запустить тесты.

ИИ-ассистента полезно просить обновлять документацию после заметных изменений.

## Инфраструктура и публикация

В проекте могут находиться файлы, связанные с размещением сайта.

Они описывают:

* как собрать приложение;
* как запустить сервер;
* какие ресурсы нужны;
* как создать контейнер;
* как настроить автоматическую публикацию;
* как подключить домен;
* какие переменные нужны.

К инфраструктуре могут относиться:

* конфигурация хостинга;
* Docker-файлы;
* настройки CI/CD;
* серверные конфигурации;
* сценарии развертывания.

Новичку не обязательно сразу разбираться во всех этих технологиях. Важно понимать, что публикация проекта также является частью его структуры.

## Пример структуры простого сайта

Простой информационный сайт может выглядеть так:

```text
project/
├── index.html
├── about.html
├── css/
│   └── styles.css
├── js/
│   └── main.js
├── images/
│   ├── logo.png
│   └── hero.jpg
└── README.md
```

Здесь:

* HTML-файлы содержат страницы;
* папка `css` содержит стили;
* папка `js` содержит поведение;
* папка `images` хранит изображения;
* README объясняет проект.

Такой сайт может не иметь backend и базы данных.

## Пример структуры frontend-приложения

Условная структура современного frontend может выглядеть так:

```text
frontend/
├── public/
├── src/
│   ├── components/
│   ├── pages/
│   ├── services/
│   ├── hooks/
│   ├── utils/
│   ├── styles/
│   └── main.js
├── package.json
├── .env
└── README.md
```

Здесь:

* `components` — повторно используемые части интерфейса;
* `pages` — страницы;
* `services` — функции для работы с API;
* `hooks` — повторно используемая логика, если она поддерживается технологией;
* `utils` — вспомогательные функции;
* `styles` — стили;
* `package.json` — зависимости и команды;
* `.env` — настройки окружения.

Это только пример. В реальном проекте названия и расположение папок могут отличаться.

## Пример структуры backend

Условный backend может быть организован так:

```text
backend/
├── src/
│   ├── routes/
│   ├── controllers/
│   ├── services/
│   ├── models/
│   ├── middleware/
│   ├── database/
│   ├── utils/
│   └── server.js
├── tests/
├── package.json
├── .env
└── README.md
```

Здесь:

* `routes` — адреса API;
* `controllers` — обработка запросов;
* `services` — бизнес-логика;
* `models` — описание данных;
* `middleware` — промежуточные проверки;
* `database` — подключение и работа с базой;
* `tests` — автоматические проверки;
* `server.js` — точка запуска сервера.

Не нужно считать эту структуру обязательным стандартом. Ее задача — показать разделение ответственности.

## Пример структуры fullstack-проекта

Frontend и backend могут находиться в одном репозитории:

```text
project/
├── frontend/
├── backend/
├── shared/
├── docs/
├── scripts/
├── docker/
├── README.md
└── .gitignore
```

Папка `shared` может содержать общие типы, правила или вспомогательный код.

Папка `docs` — документацию.

Папка `scripts` — команды для обслуживания проекта.

В другом fullstack-фреймворке frontend и backend могут находиться ближе друг к другу. Например, страницы, API-маршруты и серверные функции могут быть частью одной общей структуры.

## Что такое корневая папка

Корневая папка — это главный уровень проекта.

Именно в ней часто находятся:

* README;
* файл зависимостей;
* настройки сборки;
* конфигурация Git;
* папки frontend и backend;
* команды запуска.

Когда ИИ предлагает создать файл, важно уточнять путь относительно корневой папки.

Например:

> Создай файл `src/services/appointments.js`.

Так понятнее, где именно он должен находиться.

## Что такое точка входа

Точка входа — это файл, с которого начинается запуск приложения.

Для frontend она может подключать главный компонент и запускать интерфейс.

Для backend она может:

* создать сервер;
* подключить маршруты;
* загрузить настройки;
* установить соединение с базой;
* начать принимать запросы.

Название точки входа зависит от технологии.

Например:

* `main.js`;
* `index.js`;
* `app.js`;
* `server.js`.

Если проект не запускается, один из первых вопросов — правильно ли указана точка входа.

## Что такое middleware

Middleware — это промежуточный слой, через который проходит запрос до основной логики.

Он может:

* проверять авторизацию;
* записывать запрос в лог;
* проверять формат данных;
* ограничивать количество запросов;
* обрабатывать CORS;
* добавлять техническую информацию;
* перехватывать ошибки.

Например, перед открытием административного маршрута middleware может проверить, имеет ли пользователь роль администратора.

Это помогает не повторять одинаковую проверку в каждом обработчике.

## Что такое утилиты

Утилиты — это небольшие вспомогательные функции, которые используются в разных частях проекта.

Например:

* форматирование даты;
* обработка строки;
* преобразование валюты;
* создание идентификатора;
* проверка значения;
* работа с файлами.

Папка `utils` не должна превращаться в склад любой непонятной логики.

Если функция относится к конкретной бизнес-области, лучше хранить ее рядом с этой областью.

## Что такое модули и функциональные области

Большой проект можно организовать не только по типам файлов, но и по функциональным областям.

Например:

```text
src/
├── appointments/
│   ├── components/
│   ├── api/
│   ├── services/
│   └── validation/
├── users/
├── doctors/
└── payments/
```

Здесь код группируется по функциям проекта:

* записи;
* пользователи;
* врачи;
* платежи.

Такой подход помогает видеть все элементы одной функции рядом.

Нет универсального ответа, какая структура лучше. Для небольшого проекта проще разделение по типам файлов, а в крупном проекте удобнее группировка по функциональным областям.

## Почему важно разделение ответственности

Разделение ответственности означает, что каждая часть кода решает понятную задачу.

Например:

* компонент показывает форму;
* функция API отправляет запрос;
* backend-маршрут принимает его;
* сервис проверяет правила;
* модель работает с данными;
* база хранит результат.

Если все это находится в одном файле, код становится сложнее:

* читать;
* изменять;
* тестировать;
* объяснять ИИ;
* проверять после изменений.

Понятное разделение снижает риск, что изменение одной функции случайно сломает другую.

## Как одна функция проходит через структуру проекта

Рассмотрим добавление записи на прием.

### На frontend

Могут понадобиться:

* страница записи;
* форма;
* календарь;
* проверка полей;
* функция отправки запроса;
* отображение ошибки и успеха.

### На backend

Могут понадобиться:

* маршрут POST;
* контроллер;
* сервис создания записи;
* проверка доступного времени;
* проверка пользователя;
* обработка ошибок.

### В базе данных

Могут понадобиться:

* таблица или коллекция записей;
* связь с пациентом;
* связь с врачом;
* статус;
* ограничения.

### В инфраструктуре

Могут понадобиться:

* переменные окружения;
* почтовый сервис;
* фоновая отправка уведомления;
* настройки публикации.

Одна функция может затронуть множество файлов. Поэтому просьба «добавь запись» без контекста часто приводит к неполному решению.

## Почему это важно при работе с ИИ-ассистентом

ИИ работает лучше, когда понимает структуру проекта.

Перед задачей полезно сообщить:

* используемые технологии;
* основные папки;
* где находится frontend;
* где находится backend;
* как работает база данных;
* где настроены маршруты;
* как выполняются запросы;
* какие файлы относятся к нужной функции.

Например:

> Проект состоит из React-frontend в папке `frontend` и Node.js-backend в папке `backend`. API-запросы frontend находятся в `frontend/src/services`, маршруты backend — в `backend/src/routes`, бизнес-логика — в `backend/src/services`. Добавь функцию отмены записи, сохранив текущее разделение ответственности.

Такой промт снижает вероятность, что ИИ создаст случайные файлы или поместит логику не туда.

## Как дать ИИ структуру проекта

Можно передать ИИ дерево папок.

Например:

```text
src/
├── components/
├── pages/
├── services/
├── utils/
└── main.js
```

После этого полезно попросить:

* объяснить назначение папок;
* определить, где должна находиться новая функция;
* найти нарушение структуры;
* предложить минимальное изменение;
* не создавать новые папки без необходимости.

Хороший запрос:

> Ниже структура проекта. Сначала объясни, какие файлы нужно изменить для новой функции. Не пиши код, пока не определишь frontend, backend и слой данных.

Так человек сохраняет контроль над архитектурой до начала генерации кода.

## Как просить ИИ изучить незнакомый проект

Полезный порядок:

1. Передать дерево папок.
2. Показать README.
3. Показать файл зависимостей.
4. Показать точки входа.
5. Показать ключевые маршруты.
6. Показать файлы нужной функции.
7. Попросить описать связи.
8. Только потом просить изменения.

Можно использовать промт:

> Проанализируй структуру проекта и составь карту: где frontend, backend, база данных, API, конфигурация и точка запуска. Отдельно укажи, какие файлы относятся к функции записи на прием.

Это надежнее, чем сразу просить ИИ менять код.

## Как проверять файлы, созданные ИИ

После ответа ИИ стоит проверить:

* соответствует ли путь существующей структуре;
* не создан ли дублирующий файл;
* не перенесена ли серверная логика во frontend;
* не смешана ли работа с API и отображение интерфейса;
* используются ли существующие функции;
* обновлены ли импорты;
* добавлены ли новые зависимости;
* нужны ли изменения в конфигурации;
* обновлены ли тесты;
* описаны ли переменные окружения;
* обновлена ли документация.

ИИ может предложить рабочий код, который плохо вписывается в проект. Поэтому важно оценивать не только отдельный файл, но и его место в общей структуре.

## Как не дать проекту усложниться

ИИ-ассистент может создавать новые слои, папки и абстракции даже для небольшой задачи.

Перед изменением полезно установить ограничения:

* использовать текущую структуру;
* не добавлять новые библиотеки без необходимости;
* не создавать новый сервис, если уже есть подходящий;
* не переносить весь проект ради одной функции;
* сначала предложить минимальный план;
* объяснить назначение каждого нового файла.

Например:

> Реализуй изменение с минимальным количеством новых файлов. Используй существующую архитектуру и объясни, зачем нужен каждый созданный файл.

Это помогает не превращать простой проект в слишком сложную систему.

## Простая аналогия

Структуру веб-проекта можно сравнить со зданием клиники.

* Frontend — приемная и кабинеты, которые видит посетитель.
* Backend — внутренние рабочие процессы.
* База данных — архив и учетная система.
* API — правила передачи информации между отделами.
* Компоненты — типовые элементы мебели и оборудования.
* Конфигурация — инженерные настройки здания.
* Переменные окружения — закрытые служебные данные.
* Тесты — проверки оборудования и рабочих процессов.
* README — инструкция для сотрудников.
* Инфраструктура — электричество, связь, охрана и обслуживание.

Если все документы, оборудование и процессы сложить в одной комнате, работать будет трудно.

Так же и в коде: понятная структура помогает быстро находить нужную часть и безопасно вносить изменения.

## Частые заблуждения новичков

### «Сайт — это один HTML-файл»

Так может выглядеть только очень простой проект. Современный сайт обычно включает стили, скрипты, изображения, настройки и другие части.

### «Названия папок всегда одинаковые»

Разные технологии используют разные соглашения. Важнее назначение папки, а не ее название.

### «Frontend, backend и база должны находиться в одном проекте»

Они могут храниться вместе или в отдельных репозиториях и сервисах.

### «Чем больше папок, тем лучше архитектура»

Избыточное разделение усложняет проект. Структура должна соответствовать его масштабу.

### «Вся логика должна быть в компонентах»

Компоненты отвечают прежде всего за интерфейс. Запросы, бизнес-правила и общие функции часто лучше отделять.

### «Папка utils подходит для любого кода»

Она предназначена для общих вспомогательных функций, а не для случайной бизнес-логики.

### «Если ИИ создал файл, проект сразу начнет его использовать»

Новый файл нужно подключить через импорт, маршрут, конфигурацию или точку входа.

### «Изменение модели автоматически меняет базу данных»

Для реальной базы может потребоваться миграция.

### «Файл .env можно отправить ИИ целиком»

Он может содержать секреты. Лучше передавать только названия переменных и обезличенные примеры.

### «ИИ сам разберется в структуре без контекста»

Без дерева папок и ключевых файлов он может придумать архитектуру, которой нет в проекте.

## Что важно запомнить

* Веб-проект состоит из нескольких связанных частей, а не из одного файла.
* Frontend отвечает за интерфейс и работу в браузере.
* Backend отвечает за серверную логику и обработку запросов.
* База данных хранит информацию проекта.
* Статические файлы включают изображения, шрифты и документы.
* Конфигурация определяет запуск, сборку и поведение проекта.
* Переменные окружения хранят настройки и секретные значения.
* Зависимости подключают готовые библиотеки.
* Тесты помогают проверять работу функций.
* README объясняет, как устроен и запускается проект.
* Понятная структура строится на разделении ответственности.
* Одна функция может затрагивать frontend, backend, базу и инфраструктуру.
* Перед изменениями полезно показать ИИ дерево папок и ключевые файлы.
* Человек должен контролировать архитектуру и место каждого изменения.

## Промты для самостоятельного уточнения

* «Объясни простыми словами, из каких основных частей состоит современный веб-проект».
* «Проанализируй дерево папок моего проекта и объясни назначение каждой основной директории».
* «Покажи, где в веб-проекте обычно находятся страницы, компоненты, API-запросы, backend и работа с базой данных».
* «Помоги определить, какие файлы нужно изменить для добавления новой функции».
* «Проверь структуру моего проекта и найди места, где смешаны интерфейс, бизнес-логика и работа с данными».
* «Предложи простую структуру для небольшого fullstack-проекта без лишних папок и сложной архитектуры».
* «Объясни, какие файлы проекта нельзя публиковать и где должны храниться секретные настройки».
* «Составь промт для ИИ-ассистента, чтобы он сначала изучил структуру проекта, а затем предложил минимальные изменения».
