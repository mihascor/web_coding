# Маршруты и endpoints: куда отправляются запросы

## Коротко

Когда frontend отправляет запрос на сервер, ему недостаточно просто указать адрес сайта. Сервер должен понять, какое именно действие требуется: получить список товаров, создать запись, изменить профиль пользователя или удалить комментарий.

Для этого в backend используются **маршруты** и **endpoints**. Они помогают направлять запросы в нужную часть серверного приложения.

Понимание маршрутов особенно важно при работе с ИИ-ассистентом. Оно позволяет точнее описывать, какие действия должен поддерживать backend, какие данные он принимает и что возвращает в ответ.

## Что такое маршрут

**Маршрут**, или route, — это правило, по которому сервер определяет, что нужно сделать с поступившим запросом.

Обычно маршрут учитывает две основные части:

* адрес запроса;
* HTTP-метод запроса.

Адрес может выглядеть так:

`/users`

`/appointments`

`/products`

`/orders/125`

Он показывает, с каким ресурсом или разделом приложения связан запрос.

HTTP-метод обозначает действие. Например, один запрос может получать данные, другой — создавать новые, третий — изменять или удалять существующие.

Поэтому один и тот же адрес может использоваться для разных действий.

Например:

* получить список записей;
* создать новую запись;
* изменить конкретную запись;
* удалить запись.

Сервер различает эти действия по маршруту и типу запроса.

## Что такое endpoint

**Endpoint** — это конкретная точка взаимодействия с сервером, через которую клиент может выполнить определённое действие.

В повседневной веб-разработке слова «маршрут» и «endpoint» часто используются почти как синонимы. Но между ними можно провести небольшое различие.

Маршрут — это правило внутри backend, которое направляет запрос к нужной обработке.

Endpoint — это доступная клиенту точка API, состоящая из адреса и способа обращения к нему.

Например, endpoint может означать:

* получить список услуг;
* создать заказ;
* получить информацию о пользователе;
* изменить статус записи;
* удалить товар из корзины.

То есть endpoint описывает не просто адрес, а конкретную возможность, которую сервер предоставляет frontend или другому приложению.

## Для чего нужны маршруты и endpoints

Backend может выполнять десятки или сотни разных операций.

Без маршрутов сервер не понимал бы, что именно требуется сделать с каждым новым запросом.

Маршруты помогают разделить серверные функции по смыслу:

* запросы пользователей;
* работа с заказами;
* управление услугами;
* регистрация и вход;
* загрузка файлов;
* отправка уведомлений;
* работа административной панели.

Например, запрос на получение списка врачей должен обрабатываться не так, как запрос на изменение пароля пользователя. У них разные адреса, данные, проверки и результаты.

Endpoints создают понятный интерфейс для взаимодействия с backend. Frontend не должен знать, как именно сервер выполняет действие внутри. Ему достаточно знать:

* куда отправить запрос;
* каким способом его отправить;
* какие данные передать;
* какой ответ ожидать.

## Из чего обычно состоит endpoint

Endpoint можно представить как сочетание нескольких элементов.

### Адрес

Адрес показывает, к какому ресурсу относится запрос.

Например:

`/users`

`/orders`

`/appointments`

`/services`

Адреса стараются делать понятными, чтобы по ним можно было предположить назначение запроса.

### HTTP-метод

HTTP-метод сообщает, какое действие требуется выполнить.

На базовом уровне достаточно понимать четыре распространённых действия:

* получить данные;
* создать новые данные;
* изменить существующие данные;
* удалить данные.

Эти действия часто связывают с методами GET, POST, PUT, PATCH и DELETE.

Тебе не обязательно сразу запоминать все технические различия между ними. Важно понимать сам принцип: адрес указывает на объект, а метод — на действие с этим объектом.

### Параметры

Иногда серверу нужно понять, с каким конкретным объектом ведётся работа.

Например:

`/users/42`

Число в конце может обозначать идентификатор пользователя.

Другой пример:

`/orders/125`

Такой адрес может относиться к заказу с определённым номером.

### Данные запроса

При создании или изменении информации frontend обычно передаёт серверу данные.

Например, при создании записи это могут быть:

* имя клиента;
* выбранная услуга;
* специалист;
* дата;
* время.

Endpoint определяет, какие данные он ожидает получить и какие из них обязательны.

### Ответ сервера

После обработки запроса endpoint возвращает результат.

В ответе могут находиться:

* запрошенные данные;
* подтверждение успешного действия;
* созданный объект;
* сообщение об ошибке;
* информация о неправильных данных;
* указание на отсутствие прав доступа.

## Как сервер выбирает нужный маршрут

Когда запрос приходит на backend, сервер анализирует его адрес и HTTP-метод.

Представим, что приложение получает запрос, связанный с заказами.

Сервер определяет:

1. К какому разделу относится адрес.
2. Какое действие обозначает метод запроса.
3. Есть ли в адресе идентификатор конкретного объекта.
4. Какие данные передал клиент.
5. Какой обработчик должен выполнить задачу.

После этого запрос направляется в нужную часть backend.

Там могут выполняться проверки, бизнес-логика, работа с базой данных и обращение к внешним сервисам.

Маршрут сам по себе обычно не содержит всю логику приложения. Его задача — принять запрос и передать его нужной функции или группе функций.

## Как маршруты связаны с frontend

Frontend отправляет запросы к endpoints, когда пользователю нужны данные или когда он выполняет действие.

Например, при открытии страницы каталога frontend может запросить список товаров.

Когда пользователь оформляет заказ, frontend отправляет данные заказа на другой endpoint.

Когда пользователь открывает свой профиль, frontend обращается к endpoint, который возвращает информацию об аккаунте.

Таким образом, frontend и backend заранее договариваются о правилах взаимодействия:

* какие endpoints существуют;
* какие данные они принимают;
* что возвращают;
* какие ошибки возможны.

Эту договорённость иногда называют **контрактом API**.

Если frontend ожидает одни данные, а backend возвращает другие, приложение может работать неправильно, даже если обе части по отдельности не содержат явных ошибок.

## Как маршруты связаны с базой данных

Endpoint не является базой данных и обычно не работает с ней напрямую без промежуточной логики.

Запрос проходит несколько этапов:

1. Маршрут принимает запрос.
2. Backend проверяет данные и права пользователя.
3. Бизнес-логика определяет, что нужно сделать.
4. Сервер обращается к базе данных.
5. Полученный результат преобразуется в ответ.
6. Ответ возвращается frontend.

Например, endpoint для создания записи может проверить доступность времени, найти нужного специалиста, создать запись в базе и только после этого сообщить frontend об успехе.

## Параметры в адресе и параметры запроса

Дополнительную информацию можно передавать серверу разными способами.

### Параметры пути

Они являются частью адреса и обычно указывают на конкретный объект.

Например:

`/users/42`

`/appointments/815`

Такие параметры отвечают на вопрос: «С каким именно объектом мы работаем?»

### Параметры фильтрации

Они помогают уточнить, какие данные нужно получить.

Например, frontend может запросить:

* записи на определённую дату;
* товары конкретной категории;
* заказы с выбранным статусом;
* первые двадцать результатов;
* данные, отсортированные по цене.

В этом случае endpoint остаётся тем же, но к запросу добавляются условия поиска, фильтрации или сортировки.

На базовом уровне важно понимать: адрес может указывать не только на раздел приложения, но и на конкретный объект или выбранный набор данных.

## Закрытые и открытые endpoints

Не каждый endpoint должен быть доступен всем пользователям.

Некоторые endpoints могут быть открытыми:

* получение списка услуг;
* просмотр публичных материалов;
* отправка формы обратной связи;
* регистрация пользователя.

Другие должны быть защищены:

* просмотр личных данных;
* изменение профиля;
* управление заказами;
* доступ к административной панели;
* изменение цен и услуг.

Перед выполнением защищённого запроса backend проверяет, кто его отправил и имеет ли этот пользователь необходимые права.

Поэтому при описании endpoint важно указывать не только его назначение, но и доступные роли.

## Почему это важно при работе с ИИ-ассистентом

Если попросить ИИ «сделать API для записей», задача будет слишком общей.

Ассистенту придётся самостоятельно предполагать:

* какие действия нужны;
* какие endpoints создать;
* какие данные они принимают;
* кому разрешён доступ;
* какие проверки выполнять;
* что возвращать при успехе или ошибке.

Часть предположений может не совпасть с логикой твоего проекта.

Более точная постановка задачи может включать:

* список необходимых действий;
* названия ресурсов;
* правила доступа;
* обязательные данные;
* варианты ответа;
* возможные ошибки;
* связь с базой данных.

Например, можно попросить ИИ сначала составить список endpoints для системы онлайн-записи и объяснить назначение каждого из них без написания кода.

После этого ты сможешь проверить структуру и только затем переходить к реализации.

Также полезно попросить ИИ создать таблицу API-контракта, где для каждого endpoint будут указаны:

* действие;
* адрес;
* HTTP-метод;
* входные данные;
* права доступа;
* успешный ответ;
* возможные ошибки.

Такой подход помогает сохранить контроль над архитектурой проекта.

## Простая аналогия

Маршруты можно сравнить с отделами в большой организации.

Если отправить письмо просто «в компанию», непонятно, кто должен его обработать. Поэтому письмо направляют в бухгалтерию, службу поддержки, отдел продаж или отдел кадров.

Адрес маршрута показывает нужный отдел, а тип запроса — какое действие требуется выполнить.

Например, одно обращение может запрашивать информацию, другое — создавать заявку, третье — изменять существующие данные.

Endpoint в этой аналогии похож на конкретное окно обслуживания с понятными правилами: куда обратиться, какие документы предоставить и какой результат получить.

## Частые заблуждения новичков

### Endpoint — это только адрес

Конкретный endpoint обычно определяется не только адресом, но и HTTP-методом. Один адрес может поддерживать несколько разных действий.

### Каждый endpoint должен иметь уникальный адрес

Один адрес может использоваться для получения и создания данных. Сервер различает действия по HTTP-методу.

### Маршрут должен содержать всю бизнес-логику

Маршрут чаще всего только принимает запрос и направляет его дальше. Сложные правила лучше хранить в отдельных частях backend.

### Frontend может отправить любой набор данных

Backend ожидает определённую структуру запроса. Неизвестные, неправильные или неполные данные должны быть проверены сервером.

### Название endpoint не имеет значения

Непонятные и несогласованные адреса усложняют проект. Имена маршрутов должны отражать ресурсы и действия приложения.

### Все endpoints должны быть публичными

Часть endpoints должна быть доступна только авторизованным пользователям или определённым ролям.

### ИИ самостоятельно создаст правильную структуру API

ИИ может предложить типовую структуру, но не знает всех правил проекта. Список действий, роли пользователей и ограничения должен определить человек.

## Что важно запомнить

* Маршрут помогает backend определить, как обработать запрос.
* Endpoint — это конкретная точка взаимодействия с сервером.
* Endpoint обычно включает адрес и HTTP-метод.
* Адрес показывает ресурс, а метод — действие с ним.
* В запросе могут передаваться идентификаторы, фильтры и другие данные.
* Frontend и backend должны одинаково понимать структуру запросов и ответов.
* Не все endpoints должны быть доступны всем пользователям.
* Маршруты не должны превращаться в место, где хранится вся логика приложения.
* При работе с ИИ полезно сначала согласовать список endpoints и их назначение, а затем просить писать код.

## Промты для самостоятельного уточнения

* «Объясни простыми словами разницу между маршрутом, endpoint и API.»

* «Составь список endpoints для моего веб-приложения и кратко объясни назначение каждого из них.»

* «Покажи без сложного кода, как frontend отправляет запрос на endpoint и получает ответ от backend.»

* «Помоги определить, какие HTTP-методы подходят для получения, создания, изменения и удаления данных.»

* «Составь таблицу API-контракта с адресами, методами, входными данными, ответами и возможными ошибками.»

* «Проанализируй мои endpoints и укажи, какие из них должны быть публичными, а какие защищёнными.»

* «Объясни, как параметры пути, фильтры и данные запроса используются в одном API.»

* «Проверь структуру маршрутов моего проекта и предложи более понятные названия без изменения бизнес-логики.»
