# Компонентный подход: из чего собираются современные интерфейсы

## Коротко

Современный сайт редко создаётся как одна большая страница, в которой все элементы тесно связаны друг с другом. Обычно интерфейс собирают из отдельных частей — компонентов.

Кнопка, карточка товара, форма записи, меню, окно уведомления и шапка сайта могут быть самостоятельными компонентами. Их можно настраивать, повторно использовать и объединять в более крупные блоки.

Понимание компонентного подхода помогает правильно описывать структуру интерфейса ИИ-ассистенту. Вместо просьбы «сделай страницу» ты можешь объяснить, из каких частей она должна состоять и как эти части должны взаимодействовать.

## Что такое компонент

Компонент — это самостоятельная часть пользовательского интерфейса, которая выполняет определённую задачу.

Примеры компонентов:

* кнопка;
* поле ввода;
* меню;
* карточка услуги;
* форма обратной связи;
* календарь;
* список товаров;
* всплывающее окно;
* шапка сайта;
* боковая панель;
* блок отзывов.

Компонент может быть совсем небольшим, например иконкой или переключателем. Он также может быть крупным и включать другие компоненты.

Например, карточка услуги может содержать:

* изображение;
* название услуги;
* описание;
* цену;
* кнопку записи.

Каждый из этих элементов может быть отдельным компонентом, а вместе они образуют более крупный компонент — карточку услуги.

## Для чего используется компонентный подход

Компонентный подход помогает разделить большой интерфейс на понятные и управляемые части.

Это даёт несколько преимуществ.

### Повторное использование

Один компонент можно применять в разных местах проекта.

Например, одна и та же кнопка может использоваться в форме записи, личном кабинете и каталоге услуг. Её не нужно создавать заново для каждой страницы.

### Единообразие интерфейса

Если все похожие элементы создаются на основе одного компонента, они выглядят и работают одинаково.

Изменение основного компонента кнопки может обновить кнопки сразу на нескольких страницах.

### Удобство изменений

Небольшую часть интерфейса проще изменить, чем искать нужный элемент внутри одной большой страницы.

Например, можно переработать карточку специалиста, не затрагивая меню, форму записи и другие части сайта.

### Понятная структура проекта

Компоненты помогают разработчику и ИИ-ассистенту понимать, где находится определённая часть интерфейса и за что она отвечает.

## Как компонент может меняться

Один компонент не обязательно всегда выглядит одинаково. Ему можно передавать разные данные и настройки.

Например, компонент карточки специалиста может получать:

* имя;
* фотографию;
* специализацию;
* стоимость приёма;
* доступное время;
* текст кнопки.

Структура карточки остаётся общей, но содержимое меняется.

Компонент кнопки может иметь разные варианты:

* основная кнопка;
* второстепенная кнопка;
* кнопка удаления;
* кнопка с иконкой;
* неактивная кнопка.

Это позволяет не создавать отдельный компонент для каждого похожего элемента.

В современных frontend-инструментах данные и настройки, которые передаются компоненту, часто называют свойствами или `props`. На базовом уровне достаточно понимать, что они помогают настраивать компонент снаружи.

## Из чего собирается страница

Страницу можно представить как дерево компонентов.

Например, страница клиники может состоять из следующих крупных частей:

1. Шапка сайта.
2. Главное меню.
3. Информация о клинике.
4. Список услуг.
5. Карточки специалистов.
6. Форма записи.
7. Отзывы.
8. Подвал сайта.

При этом список услуг состоит из карточек услуг, форма записи — из полей и кнопок, а меню — из отдельных пунктов.

Получается несколько уровней:

* страница;
* крупные разделы страницы;
* компоненты внутри разделов;
* небольшие элементы внутри компонентов.

Такой подход называют композицией: сложный интерфейс собирается из более простых частей.

## Как компоненты связаны с HTML, CSS и JavaScript

Компонентный подход не заменяет основные технологии веб-разработки.

Внутри компонента по-прежнему используются:

* HTML для структуры;
* CSS для оформления;
* JavaScript для поведения и обработки действий пользователя.

Отличие заключается в организации проекта. Код, оформление и поведение конкретной части интерфейса объединяются вокруг одной понятной задачи.

Например, компонент формы записи может отвечать за:

* отображение полей;
* получение введённых данных;
* проверку заполнения;
* показ ошибок;
* отправку данных;
* отображение результата.

Компоненты особенно часто используются во frontend-фреймворках и библиотеках, таких как React, Vue и Angular. Эти инструменты помогают собирать интерфейс из компонентов и управлять их обновлением.

## Компоненты и состояние

Некоторые компоненты только показывают информацию. Другие должны помнить, что происходит во время взаимодействия с пользователем.

Например, компонент может хранить:

* открыто или закрыто меню;
* выбран ли переключатель;
* какое значение введено в поле;
* какой пункт списка выбран;
* загружаются ли данные;
* произошла ли ошибка.

Такая изменяемая информация называется состоянием компонента.

Когда состояние меняется, интерфейс тоже может измениться. Например, после нажатия кнопки меню открывается, а после отправки формы появляется сообщение об успешной записи.

Подробно разбирать управление состоянием на этом этапе не обязательно. Важно понимать, что компонент может не только отображать данные, но и реагировать на действия пользователя.

## Как определить границы компонента

Не каждый отдельный текст или отступ обязательно нужно превращать в самостоятельный компонент.

Обычно отдельный компонент имеет смысл выделять, если он:

* используется в нескольких местах;
* выполняет самостоятельную задачу;
* имеет собственное поведение;
* получает данные;
* может изменяться независимо от остальной страницы;
* является понятной частью интерфейса.

Например, повторяющуюся карточку товара логично сделать компонентом. Обычный абзац внутри одной страницы чаще всего не требует отдельного компонента.

Слишком крупные компоненты сложно изменять и понимать. Слишком большое количество мелких компонентов также может запутать структуру проекта.

Поэтому интерфейс нужно делить на части осмысленно, исходя из их назначения.

## Компонентные библиотеки

В проекте не всегда нужно создавать все компоненты с нуля.

Существуют готовые компонентные библиотеки. Они могут содержать:

* кнопки;
* формы;
* таблицы;
* меню;
* уведомления;
* календари;
* модальные окна;
* элементы навигации.

Такие библиотеки ускоряют разработку и помогают сохранять единый стиль интерфейса.

Однако готовый компонент может иметь ограничения. Его внешний вид, поведение или структура не всегда подходят конкретному проекту. Поэтому перед использованием важно понимать, насколько легко его можно настроить.

## Почему это важно при работе с ИИ-ассистентом

ИИ-ассистент лучше справляется с задачей, когда получает понятную структуру интерфейса.

Запрос «создай страницу личного кабинета» оставляет слишком много решений на усмотрение ИИ. Ассистент сам определит состав страницы, расположение блоков и связи между ними.

Более точная задача может включать:

* список необходимых компонентов;
* назначение каждого компонента;
* данные, которые он должен отображать;
* действия пользователя;
* варианты состояния;
* правила взаимодействия компонентов;
* требования к адаптивности;
* элементы, которые будут использоваться повторно.

Например, можно попросить ИИ создать страницу из бокового меню, блока профиля, истории записей и карточек предстоящих посещений.

Компонентный подход также помогает давать задачи поэтапно. Сначала можно создать общую структуру страницы, затем разработать отдельные компоненты и после этого объединить их.

Так проще проверять результат и исправлять ошибки.

## Простая аналогия

Компонентный интерфейс можно сравнить с конструктором.

В конструкторе есть отдельные детали: колёса, двери, окна и основные блоки. Из одинаковых деталей можно собрать разные объекты.

Компоненты работают похожим образом. Одна кнопка может использоваться на разных страницах, карточки могут получать разное содержимое, а из готовых блоков можно собирать новые интерфейсы.

При этом важно не только иметь набор деталей, но и понимать, как они должны соединяться и какую общую задачу выполняет готовая конструкция.

## Частые заблуждения новичков

### Компонент — это любая HTML-часть страницы

Компонент обычно имеет самостоятельное назначение. Не каждый заголовок, абзац или контейнер нужно выделять в отдельный компонент.

### Чем больше компонентов, тем лучше

Слишком мелкое разделение усложняет проект. Компоненты должны помогать понимать интерфейс, а не создавать лишнюю структуру.

### Компоненты нужны только для повторного использования

Повторное использование важно, но компоненты также помогают разделять ответственность, управлять поведением и упрощать поддержку проекта.

### Каждый компонент полностью независим

Компоненты могут передавать друг другу данные и реагировать на общие события. Главное — делать эти связи понятными и контролируемыми.

### Компонентный подход существует только в React

React активно использует компоненты, но сам подход применяется и в других технологиях, включая Vue, Angular и обычную веб-разработку.

### ИИ сам правильно разделит страницу на компоненты

ИИ может предложить структуру, но не всегда понимает будущие изменения проекта и бизнес-задачи. Человек должен проверить, логично ли разделён интерфейс.

## Что важно запомнить

* Компонент — это самостоятельная часть пользовательского интерфейса.
* Современные страницы собираются из небольших и крупных компонентов.
* Один компонент можно использовать с разными данными и настройками.
* Компоненты помогают сохранять единый стиль и упрощают изменения.
* Сложные компоненты могут состоять из более простых.
* Некоторые компоненты хранят состояние и реагируют на действия пользователя.
* Не нужно превращать каждый элемент страницы в отдельный компонент.
* При работе с ИИ полезно описывать не только всю страницу, но и её составные части.
* ИИ помогает создавать компоненты, но человек определяет их назначение и связи.

## Промты для самостоятельного уточнения

* «Объясни мне на простом примере, как страница сайта разделяется на компоненты».

* «Предложи компонентную структуру для страницы, которую я опишу. Для каждого компонента кратко укажи его назначение».

* «Помоги определить, какие элементы моего интерфейса должны быть отдельными компонентами, а какие можно оставить внутри страницы».

* «Объясни простыми словами, чем компонент отличается от обычного HTML-элемента».

* «Покажи без сложного кода, как один компонент может отображать разные данные».

* «Объясни, как компоненты связаны с HTML, CSS, JavaScript и состоянием интерфейса».

* «Проверь предложенную мной структуру компонентов и найди слишком крупные, слишком мелкие или дублирующиеся части».

* «Составь промт для ИИ-ассистента, чтобы он создал страницу из отдельных переиспользуемых компонентов».
