# Fetch-запросы: как сайт получает данные с сервера

## Коротко

Современный сайт редко содержит всю необходимую информацию внутри одной страницы. Каталог товаров, расписание, профиль пользователя, новости, цены и результаты поиска обычно загружаются с сервера.

Чтобы запросить эти данные, frontend обращается к API. В JavaScript для такого обращения часто используется инструмент `fetch`.

Тебе не обязательно сразу разбираться во всех деталях его синтаксиса. Важно понимать общую последовательность: сайт отправляет запрос, ждёт ответ, обрабатывает полученные данные и обновляет интерфейс.

## Что такое fetch-запрос

`fetch` — это встроенный в браузер инструмент JavaScript для отправки сетевых запросов.

С его помощью frontend может:

* получить данные с сервера;
* отправить данные формы;
* создать новую запись;
* изменить существующую информацию;
* удалить данные;
* обратиться к внешнему сервису через API.

Например, когда пользователь открывает страницу услуг, сайт может отправить запрос серверу:

> Передай список доступных услуг.

Сервер обрабатывает запрос и возвращает ответ, часто в формате JSON. После этого frontend показывает полученные услуги на странице.

Сам `fetch` не создаёт карточки, таблицы или кнопки. Его задача — организовать обмен данными между браузером и сервером.

## Для чего fetch-запросы используются

Fetch-запросы нужны, когда веб-страница должна получить или передать актуальную информацию без полной перезагрузки.

Они могут использоваться для:

* загрузки каталога товаров или услуг;
* получения профиля пользователя;
* отображения расписания;
* отправки формы записи;
* поиска по сайту;
* фильтрации списка;
* добавления товара в корзину;
* сохранения настроек;
* загрузки комментариев;
* получения статуса заказа.

Например, пользователь выбирает нужную дату в форме записи. Сайт отправляет серверу выбранную дату и получает список свободного времени. Страница при этом не закрывается и не загружается заново — обновляется только нужная часть интерфейса.

Именно такие фоновые обращения делают веб-приложение интерактивным.

## Как это работает в общей картине

Упрощённая последовательность fetch-запроса выглядит так:

1. Пользователь открывает страницу или выполняет действие.
2. JavaScript запускает fetch-запрос.
3. Браузер отправляет HTTP-запрос на определённый адрес API.
4. Сервер принимает запрос.
5. Backend выполняет необходимую логику.
6. При необходимости сервер обращается к базе данных.
7. Сервер формирует HTTP-ответ.
8. Браузер получает ответ.
9. JavaScript обрабатывает данные.
10. Интерфейс обновляется.

Например, пользователь открывает личный кабинет.

Frontend может запросить у сервера:

* имя пользователя;
* контактные данные;
* историю записей;
* активные заказы;
* доступные бонусы.

Backend проверяет, имеет ли пользователь право получить эту информацию, находит данные в базе и возвращает результат. Frontend распределяет полученные сведения по нужным компонентам страницы.

Таким образом, fetch-запрос находится между интерфейсом и API. Он помогает frontend обратиться к серверу и получить результат.

## Из чего состоит запрос

Чтобы сервер правильно понял обращение, запрос обычно содержит несколько частей.

### Адрес

Запрос отправляется на определённый адрес API — endpoint.

Например:

* адрес списка услуг;
* адрес конкретного пользователя;
* адрес создания записи;
* адрес удаления товара из корзины.

Endpoint сообщает серверу, с каким ресурсом хочет работать frontend.

### Метод запроса

Метод описывает предполагаемое действие.

Чаще всего используются:

* `GET` — получить данные;
* `POST` — отправить или создать данные;
* `PUT` или `PATCH` — изменить данные;
* `DELETE` — удалить данные.

Например, получение списка услуг обычно связано с `GET`, а создание новой записи на приём — с `POST`.

Это общие правила, но точное поведение определяет конкретный API.

### Заголовки

Заголовки содержат дополнительную информацию о запросе.

Они могут сообщать:

* в каком формате отправляются данные;
* какой формат ответа ожидается;
* кто выполняет запрос;
* какой язык интерфейса используется;
* какие правила обработки нужно применить.

Например, заголовок может указывать, что тело запроса передаётся в формате JSON.

### Параметры

Параметры помогают уточнить запрос.

Например, сайт может запросить:

* только услуги определённой категории;
* записи за конкретную дату;
* вторую страницу каталога;
* товары дешевле определённой суммы;
* результаты поиска по слову.

Параметры позволяют не загружать все данные сразу, а получать только необходимую часть.

### Тело запроса

Тело содержит данные, которые frontend отправляет серверу.

Оно часто используется в запросах на создание или изменение информации.

Например, при создании записи могут отправляться:

* идентификатор услуги;
* дата;
* время;
* имя клиента;
* номер телефона.

Эти данные нередко передаются в формате JSON.

## Что возвращает сервер

Сервер возвращает не только данные, но и информацию о результате обработки запроса.

Ответ обычно включает:

* статус запроса;
* заголовки;
* тело ответа.

В теле могут находиться:

* данные в формате JSON;
* сообщение об ошибке;
* подтверждение выполненного действия;
* пустой ответ;
* файл или изображение.

Например, после отправки формы сервер может вернуть:

```json
{
  "success": true,
  "appointmentId": 458
}
```

Если данные формы заполнены неправильно, ответ может быть другим:

```json
{
  "success": false,
  "message": "Выбранное время уже занято"
}
```

Frontend должен уметь обработать оба варианта и показать пользователю понятный результат.

## Почему запрос выполняется не мгновенно

Fetch-запрос связан с передачей данных по сети. Ответ может прийти быстро, медленно или не прийти вообще.

На скорость влияют:

* качество интернет-соединения;
* расстояние до сервера;
* нагрузка на сервер;
* сложность обработки данных;
* скорость базы данных;
* объём ответа;
* работа внешних сервисов.

Поэтому JavaScript не должен останавливать весь интерфейс и просто ждать завершения запроса. Запрос выполняется асинхронно.

Асинхронность означает, что браузер может продолжать выполнять другие действия, пока ожидает ответ сервера.

Пользователь в это время может видеть индикатор загрузки, продолжать читать страницу или взаимодействовать с доступными элементами.

## Состояния загрузки

При работе с сервером интерфейс обычно должен учитывать как минимум три состояния.

### Загрузка

Запрос уже отправлен, но ответ ещё не получен.

В этот момент можно показать:

* индикатор загрузки;
* временные пустые блоки;
* сообщение «Загружаем данные»;
* заблокированную кнопку отправки.

Пользователь должен понимать, что действие выполняется.

### Успех

Сервер успешно обработал запрос и вернул ожидаемый результат.

Frontend может:

* показать полученные данные;
* вывести сообщение об успешном сохранении;
* обновить список;
* закрыть форму;
* перейти на другую страницу.

### Ошибка

Запрос не удалось выполнить или сервер вернул ошибку.

Причины могут быть разными:

* отсутствует интернет;
* сервер недоступен;
* указан неправильный адрес;
* пользователь не авторизован;
* отправлены некорректные данные;
* нужная запись не найдена;
* на сервере произошла внутренняя ошибка.

Интерфейс должен не просто записать ошибку в технический журнал, а по возможности объяснить пользователю, что произошло и что делать дальше.

## Fetch и JSON

Fetch и JSON часто используются вместе, но обозначают разные вещи.

`fetch` отвечает за отправку запроса и получение ответа.

JSON отвечает за формат представления данных.

Можно представить последовательность так:

1. Frontend собирает данные.
2. Данные преобразуются в JSON.
3. Fetch отправляет их серверу.
4. Сервер возвращает JSON-ответ.
5. Frontend преобразует ответ в удобную программную структуру.
6. Интерфейс использует полученные данные.

Fetch можно сравнить с транспортом, а JSON — с форматом упаковки содержимого.

При этом fetch может работать не только с JSON. Через него можно получать текст, изображения, документы и другие типы данных.

## Fetch и API

Fetch не заменяет API.

API определяет:

* какие адреса доступны;
* какие методы можно использовать;
* какие данные нужно передать;
* какие права доступа требуются;
* что вернёт сервер.

Fetch выполняет обращение по этим правилам.

Если API ожидает идентификатор услуги и дату, frontend должен отправить именно эти данные. Если сервер возвращает список объектов с определёнными полями, интерфейс должен быть готов обработать такую структуру.

Поэтому перед созданием fetch-запроса важно понимать контракт API — согласованные правила взаимодействия frontend и backend.

## Получение данных при открытии страницы

Один из самых частых сценариев — загрузка данных при открытии страницы.

Например, пользователь переходит в раздел «Услуги». Сначала браузер отображает общую структуру страницы, а затем отправляет запрос к API.

Пока ответ не получен, пользователь видит состояние загрузки. После получения данных frontend создаёт карточки услуг.

Если сервер вернул пустой список, интерфейс может показать сообщение:

> Услуги пока не добавлены.

Если запрос завершился ошибкой:

> Не удалось загрузить услуги. Попробуй обновить страницу.

Это разные ситуации, и интерфейс должен различать их:

* данные загружаются;
* данные успешно получены;
* список пуст;
* произошла ошибка.

## Отправка данных на сервер

Fetch используется не только для получения информации, но и для её отправки.

Например, пользователь заполняет форму записи:

1. Frontend получает значения полей.
2. Проверяет их на базовом уровне.
3. Формирует структуру данных.
4. Отправляет запрос серверу.
5. Сервер повторно проверяет данные.
6. Создаёт запись в базе.
7. Возвращает результат.
8. Frontend показывает сообщение пользователю.

Важно понимать, что отправка запроса ещё не означает успешное выполнение действия.

Кнопка могла быть нажата, но сервер может отклонить данные. Поэтому результат нужно определять по ответу сервера, а не только по факту запуска fetch.

## Ошибки при работе с запросами

Ошибки можно условно разделить на несколько групп.

### Сетевая ошибка

Браузер не смог связаться с сервером.

Например:

* пропал интернет;
* сервер выключен;
* адрес недоступен;
* соединение было прервано.

### Ошибка запроса

Frontend отправил данные не в том виде, который ожидает API.

Например:

* отсутствует обязательное поле;
* неправильно указан метод;
* данные переданы не в том формате;
* использовано неправильное название параметра.

### Ошибка доступа

Сервер понял запрос, но пользователь не имеет права выполнить действие.

Например, неавторизованный пользователь пытается открыть закрытую страницу или обычный клиент пытается получить функции администратора.

### Ошибка сервера

Запрос составлен правильно, но во время обработки произошла проблема на backend.

Frontend не всегда может определить точную причину такой ошибки. Обычно он показывает понятное сообщение пользователю, а подробности сохраняются в логах сервера.

## Почему это важно при работе с ИИ-ассистентом

ИИ-ассистент может быстро написать fetch-запрос, но ему нужно дать точный контекст.

Полезно сообщить:

* адрес API;
* метод запроса;
* структуру отправляемых данных;
* пример ответа;
* нужна ли авторизация;
* когда должен запускаться запрос;
* что показывать во время загрузки;
* как отображать успешный результат;
* как обрабатывать ошибки.

Слишком общий промт:

> Загрузи услуги с сервера.

оставляет много неопределённости.

Более точный вариант:

> Создай загрузку списка услуг через API при открытии страницы. Во время запроса покажи индикатор загрузки, после успеха — карточки услуг, при пустом списке — отдельное сообщение, а при ошибке — кнопку повторной попытки.

Такой промт описывает не только сам запрос, но и ожидаемое поведение интерфейса.

Понимание fetch также помогает проверить ответ ИИ. Ты можешь спросить:

* куда отправляется запрос;
* какой метод используется;
* какие данные передаются;
* как проверяется статус ответа;
* где обрабатывается ошибка;
* что произойдёт при медленном интернете;
* не запускается ли запрос несколько раз;
* не передаются ли конфиденциальные данные.

ИИ может создать технически работающий запрос, но человек должен убедиться, что он соответствует логике проекта.

## Простая аналогия

Fetch-запрос можно сравнить с заказом в ресторане.

Ты сообщаешь официанту, что хочешь получить. Официант передаёт заказ на кухню. Кухня обрабатывает его и возвращает результат.

В этой аналогии:

* посетитель — интерфейс;
* официант — fetch-запрос;
* правила оформления заказа — API;
* кухня — backend;
* склад продуктов — база данных;
* готовое блюдо — ответ сервера;
* описание заказа — передаваемые данные.

После передачи заказа результат не появляется мгновенно. Нужно дождаться ответа. Иногда блюдо готово, иногда нужного продукта нет, а иногда на кухне произошла проблема.

Интерфейс должен учитывать все эти варианты.

## Частые заблуждения новичков

### Fetch хранит данные

Нет. Fetch только передаёт запросы и получает ответы. Данные могут храниться на сервере, в базе данных или во временном состоянии приложения.

### Fetch и API — одно и то же

Нет. API задаёт правила взаимодействия, а fetch помогает frontend отправить запрос по этим правилам.

### Если запрос отправлен, действие выполнено успешно

Нет. Сервер может вернуть ошибку, отклонить данные или сообщить, что нужный объект не найден.

### Успешный сетевой ответ всегда содержит нужные данные

Нет. Ответ может быть пустым, содержать неожиданную структуру или сообщать о проблеме. Frontend должен проверять результат.

### Ошибки нужно обрабатывать только на сервере

Нет. Сервер отвечает за свою часть проверки, а frontend должен корректно показать пользователю загрузку, успех и ошибку.

### Достаточно обработать только успешный результат

Нет. Интернет и серверы не работают идеально. Отсутствие обработки ошибок приводит к зависшим кнопкам, пустым экранам и непонятному поведению.

### Все данные нужно загружать одним запросом

Не всегда. Большие объёмы информации часто делят на страницы или загружают по мере необходимости.

### Fetch сам обновляет интерфейс

Нет. Он получает данные, но JavaScript и компоненты интерфейса должны решить, как их показать.

### Frontend может доверять любым полученным данным

Нет. Структура ответа может измениться или содержать неожиданные значения. Важные данные следует проверять перед использованием.

## Что важно запомнить

* Fetch — это инструмент JavaScript для отправки запросов из браузера.
* Он связывает frontend с API.
* Через fetch можно получать, создавать, изменять и удалять данные.
* Запрос обычно содержит адрес, метод, заголовки, параметры и иногда тело.
* Сервер возвращает статус и данные ответа.
* Запросы выполняются асинхронно и могут занимать время.
* Интерфейс должен учитывать загрузку, успех, пустой результат и ошибку.
* Fetch часто работает с JSON, но это разные понятия.
* Отправленный запрос не гарантирует успешное выполнение действия.
* ИИ-ассистенту нужно описывать не только адрес API, но и ожидаемое поведение интерфейса.
* Человек должен проверять структуру запроса, обработку ответа и возможные ошибки.
