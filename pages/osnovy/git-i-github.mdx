# Git и GitHub

## Git и GitHub: простыми словами для начинающего веб-кодера

**Git** и **GitHub** часто упоминают вместе, но это не одно и то же.

**Git** — это инструмент для контроля версий кода.  
**GitHub** — это онлайн-сервис, где можно хранить Git-проекты, работать в команде и показывать свой код другим.

Можно сказать так:

> **Git — это система управления историей проекта.**  
> **GitHub — это “облачное хранилище” и платформа для совместной работы с Git-проектами.**

---

# 1. Зачем вообще нужен Git

Когда ты пишешь сайт или приложение, проект постоянно меняется:

- добавляются новые файлы;
- исправляются ошибки;
- меняется HTML, CSS, JavaScript;
- появляются новые функции;
- иногда что-то ломается.

Без Git можно легко запутаться:

```text
project-final
project-final-2
project-final-real
project-final-real-new
project-final-new-fixed
```

Git решает эту проблему.

Он позволяет сохранять состояние проекта в разные моменты времени и при необходимости возвращаться к старым версиям.

---

# 2. Что такое контроль версий

**Контроль версий** — это система, которая следит за изменениями в файлах проекта.

Например, сегодня у тебя есть файл:

```html
<h1>Главная страница</h1>
```

Потом ты изменил его:

```html
<h1>Добро пожаловать на сайт</h1>
```

Git может запомнить:

- что именно изменилось;
- когда это произошло;
- кто это сделал;
- зачем это было сделано.

Это особенно полезно в программировании, потому что код постоянно развивается.

---

# 3. Что такое Git

**Git** — это программа, которую устанавливают на компьютер.

Она помогает:

- сохранять версии проекта;
- смотреть историю изменений;
- возвращаться к старым версиям;
- создавать отдельные ветки разработки;
- объединять изменения от разных людей;
- работать над проектом без страха что-то сломать окончательно.

Git работает локально, то есть на твоём компьютере.

---

# 4. Что такое репозиторий

**Репозиторий** — это проект, за которым следит Git.

Обычно репозиторий — это папка с файлами сайта или приложения.

Например:

```text
my-website/
├── index.html
├── style.css
├── script.js
└── images/
```

Если подключить Git к этой папке, она станет Git-репозиторием.

Git начнёт отслеживать изменения внутри проекта.

---

# 5. Что такое commit

**Commit** — это сохранённая версия проекта.

Можно представить commit как контрольную точку в игре.

Например:

```text
Commit 1: Создал HTML-структуру
Commit 2: Добавил стили
Commit 3: Подключил JavaScript
Commit 4: Исправил ошибку в меню
```

Каждый commit содержит:

- список изменений;
- автора;
- дату;
- сообщение с описанием изменений.

Сообщения обычно пишут коротко и понятно:

```text
Добавил шапку сайта
Исправил стили кнопки
Создал форму обратной связи
```

---

# 6. Что такое ветки в Git

**Ветка** — это отдельное направление разработки.

Представь, что у тебя есть рабочий сайт. Ты хочешь добавить новую функцию, но не хочешь сломать основную версию.

Тогда создаётся отдельная ветка:

```text
main
└── new-menu
```

В ветке `new-menu` можно спокойно экспериментировать. Если всё получилось хорошо, эту ветку потом объединяют с основной.

Основная ветка часто называется:

```text
main
```

Раньше часто использовалось название:

```text
master
```

---

# 7. Что такое GitHub

**GitHub** — это онлайн-платформа для хранения Git-репозиториев.

Если Git работает у тебя на компьютере, то GitHub позволяет загрузить проект в интернет.

GitHub нужен, чтобы:

- хранить код в облаке;
- делиться проектом с другими;
- работать в команде;
- смотреть историю изменений через сайт;
- принимать изменения от других разработчиков;
- размещать портфолио;
- подключать проект к хостингу или автоматической сборке.

Проще говоря:

> Git хранит историю проекта.  
> GitHub помогает хранить эту историю онлайн и работать с ней удобно.

---

# 8. Разница между Git и GitHub

| Git | GitHub |
|---|---|
| Программа на компьютере | Онлайн-сервис |
| Отслеживает изменения в коде | Хранит репозитории в интернете |
| Работает локально | Работает через сайт |
| Нужен для контроля версий | Нужен для совместной работы и публикации |
| Можно использовать без GitHub | Без Git GitHub теряет основной смысл |

Важно понимать: **GitHub — не сам Git**, а сервис, который использует Git.

Кроме GitHub есть и другие похожие сервисы:

- GitLab;
- Bitbucket;
- Gitea.

Но GitHub сейчас один из самых популярных.

---

# 9. Как выглядит обычная работа с Git и GitHub

Представим, ты создаёшь сайт.

Сначала ты работаешь у себя на компьютере:

```text
index.html
style.css
script.js
```

Потом подключаешь Git и сохраняешь изменения:

```bash
git add .
git commit -m "Создал главную страницу"
```

Затем отправляешь проект на GitHub:

```bash
git push
```

После этого проект доступен онлайн в твоём GitHub-аккаунте.

# 10. Основные команды Git

Начинающему не нужно сразу знать все команды. Достаточно понимать базовые.

## `git init`

Создаёт Git-репозиторий в папке проекта.

```bash
git init
```

## `git status`

Показывает текущее состояние проекта:

```bash
git status
```

Например, какие файлы изменились и готовы ли они к сохранению.

## `git add`

Добавляет изменения в подготовку к сохранению.

```bash
git add index.html
```

Или сразу все изменения:

```bash
git add .
```

## `git commit`

Сохраняет изменения в историю проекта.

```bash
git commit -m "Добавил главную страницу"
```

## `git log`

Показывает историю commit-ов.

```bash
git log
```

## `git clone`

Копирует репозиторий с GitHub на компьютер.

```bash
git clone ссылка-на-репозиторий
```

## `git push`

Отправляет изменения с компьютера на GitHub.

```bash
git push
```

## `git pull`

Загружает свежие изменения с GitHub на компьютер.

```bash
git pull
```

---

# 11. Что такое GitHub-репозиторий

На GitHub репозиторий выглядит как страница проекта.

Там можно увидеть:

- файлы проекта;
- историю изменений;
- README-файл;
- список веток;
- задачи;
- pull requests;
- настройки проекта.

Обычно у репозитория есть ссылка вида:

```text
github.com/username/project-name
```

Например:

```text
github.com/luda/my-first-site
```

---

# 12. Что такое README

**README** — это файл с описанием проекта.

Обычно он называется:

```text
README.md
```

В нём пишут:

- что это за проект;
- как его запустить;
- какие технологии используются;
- кто автор;
- какие функции есть;
- ссылки на демо или документацию.

Для начинающего веб-кодера README важен, потому что он помогает красиво представить проект в портфолио.

Пример простого README:

```markdown
# My First Website

Это мой первый сайт, созданный с помощью HTML, CSS и JavaScript.

## Технологии

- HTML
- CSS
- JavaScript

## Автор

Людочка
```

---

# 13. Что такое pull request

**Pull request** — это предложение внести изменения в проект.

Например, разработчик сделал новую функцию в отдельной ветке и хочет добавить её в основной проект.

Он создаёт pull request, чтобы другие участники могли:

- посмотреть изменения;
- оставить комментарии;
- предложить правки;
- одобрить или отклонить изменения.

В командной разработке pull request — очень важный инструмент.

---

# 14. Что такое merge

**Merge** — это объединение изменений из одной ветки в другую.

Например:

```text
feature-menu → main
```

Ты сделал новое меню в отдельной ветке, проверил, что всё работает, и затем объединил эту ветку с основной.

---

# 15. Что такое конфликт

**Конфликт** возникает, когда Git не может автоматически объединить изменения.

Например, два человека изменили одну и ту же строку в одном файле.

Первый написал:

```html
<h1>Добро пожаловать</h1>
```

Второй написал:

```html
<h1>Наш новый сайт</h1>
```

Git не знает, какой вариант оставить, поэтому просит человека решить конфликт вручную.

Для новичков конфликты могут казаться сложными, но это обычная часть работы с кодом.

---

# 16. Для чего Git и GitHub нужны веб-кодеру

Веб-кодеру Git и GitHub нужны почти постоянно.

Они помогают:

- хранить историю проекта;
- не бояться экспериментировать;
- быстро откатывать неудачные изменения;
- работать с другими разработчиками;
- показывать проекты работодателю или заказчику;
- загружать код на серверы и хостинги;
- вести портфолио;
- участвовать в open-source проектах.

Даже если ты работаешь одна, Git всё равно полезен.

---

# 17. GitHub как портфолио

Для начинающего разработчика GitHub часто становится частью портфолио.

Там можно показать:

- учебные проекты;
- сайты;
- приложения;
- pet-проекты;
- аккуратную историю разработки;
- умение работать с Git.

Работодатель или заказчик может открыть твой профиль и увидеть, как ты пишешь код.

Поэтому полезно держать проекты в порядке:

- давать понятные названия репозиториям;
- писать README;
- делать осмысленные commit-сообщения;
- не загружать лишние файлы;
- оформлять проекты аккуратно.

---

# 18. GitHub Pages

**GitHub Pages** — это возможность бесплатно опубликовать статический сайт прямо из GitHub-репозитория.

Это особенно удобно для начинающих, которые делают сайты на:

- HTML;
- CSS;
- JavaScript.

Например, ты создала простой сайт и загрузила его на GitHub. Через GitHub Pages можно получить ссылку и показать сайт другим людям.

Это хороший способ публиковать учебные проекты.

---

# 19. Что начинающему важно понять в первую очередь

Не нужно сразу учить Git глубоко. На старте достаточно понять базовую логику:

```text
1. Я изменила файлы.
2. Я посмотрела статус.
3. Я добавила изменения.
4. Я сделала commit.
5. Я отправила изменения на GitHub.
```

Пример:

```bash
git status
git add .
git commit -m "Добавила стили для главной страницы"
git push
```

Это базовый рабочий цикл.

---

# 20. Простая аналогия

Можно представить Git как дневник проекта.

Ты каждый день записываешь:

```text
Сегодня добавила меню.
Сегодня исправила кнопку.
Сегодня изменила цвет фона.
Сегодня подключила форму.
```

А GitHub — это онлайн-библиотека, где этот дневник и сам проект хранятся в интернете.

---

# Кратко

**Git** — это инструмент, который помогает сохранять историю изменений в проекте.

**GitHub** — это сайт, где можно хранить Git-проекты онлайн, работать с другими людьми и показывать свой код.

Для веб-кодера Git и GitHub — это почти обязательные инструменты, потому что они помогают безопасно развивать проект, работать в команде и создавать профессиональное портфолио.
