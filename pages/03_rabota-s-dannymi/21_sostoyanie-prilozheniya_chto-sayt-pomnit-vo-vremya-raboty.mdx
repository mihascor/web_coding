# Состояние приложения: что сайт «помнит» во время работы

## Коротко

Когда ты взаимодействуешь с веб-приложением, его интерфейс постоянно меняется. Открывается меню, переключается вкладка, заполняется форма, появляются результаты поиска, обновляется корзина или отображается сообщение об ошибке.

Чтобы правильно показывать текущую ситуацию, приложение должно временно «помнить», что происходит. Эти сведения называются состоянием приложения.

Состояние помогает связать данные и интерфейс. Когда состояние меняется, сайт может обновить нужные элементы страницы.

## Что такое состояние приложения

Состояние приложения — это набор данных, которые описывают текущую ситуацию в интерфейсе или приложении.

Например, сайт может помнить:

* открыто ли боковое меню;
* какая вкладка выбрана;
* что пользователь ввёл в форму;
* какие товары находятся в корзине;
* загружаются ли данные;
* произошла ли ошибка;
* какой фильтр установлен;
* какой пользователь сейчас отображается;
* выбрана ли определённая дата;
* открыто ли модальное окно.

Каждое такое значение влияет на поведение или внешний вид сайта.

Например, у меню может быть два состояния:

* закрыто;
* открыто.

У запроса к серверу состояний может быть больше:

* запрос ещё не запускался;
* данные загружаются;
* данные получены;
* список оказался пустым;
* произошла ошибка.

Состояние — это не обязательно одна переменная. В реальном веб-приложении одновременно существует множество связанных состояний.

## Как состояние связано с интерфейсом

Интерфейс показывает пользователю текущее состояние приложения.

Допустим, на странице есть кнопка «Показать фильтры». После нажатия приложение меняет состояние фильтров с «закрыты» на «открыты». Интерфейс замечает изменение и показывает панель.

Общая логика выглядит так:

1. Пользователь выполняет действие.
2. Приложение изменяет состояние.
3. Интерфейс обновляется в соответствии с новым состоянием.

Например:

* пользователь вводит текст — изменяется состояние поля;
* выбирает категорию — изменяется активный фильтр;
* добавляет товар — изменяется содержимое корзины;
* отправляет форму — состояние меняется на «отправка»;
* сервер возвращает ответ — появляется состояние успеха или ошибки.

Таким образом, пользователь обычно видит не само состояние, а результат его изменения.

## Для чего состояние используется

Состояние позволяет сайту реагировать на действия пользователя и сохранять последовательность работы внутри текущего взаимодействия.

Без состояния приложение не могло бы понять:

* что сейчас нужно показать;
* какие данные уже введены;
* какое действие выполняется;
* какой элемент выбран;
* какие данные были получены;
* нужно ли показывать загрузку;
* где произошла ошибка.

Представь форму записи на приём. Во время её заполнения приложение может хранить:

* выбранную услугу;
* выбранного специалиста;
* дату;
* время;
* имя пользователя;
* номер телефона;
* текущий шаг формы;
* наличие ошибок в полях;
* статус отправки.

Все эти данные образуют состояние формы.

Пока пользователь переходит между шагами, сайт использует состояние, чтобы не потерять уже введённую информацию.

## Какие состояния бывают

Состояние можно условно разделить на несколько видов. Это не строгая классификация, а удобный способ понять общую картину.

### Состояние интерфейса

Описывает внешний вид и поведение элементов страницы.

Например:

* открыто ли меню;
* какая вкладка активна;
* раскрыт ли список;
* показано ли уведомление;
* выбран ли элемент;
* включена ли тёмная тема.

Обычно такие данные нужны только frontend.

### Состояние формы

Содержит информацию, которую пользователь вводит или выбирает.

Например:

* текст в поле;
* выбранная дата;
* отмеченные параметры;
* ошибки валидации;
* статус отправки формы.

Часть этого состояния позже может быть отправлена серверу.

### Состояние загрузки

Показывает, что происходит при обращении к API.

Например:

* данные ещё не запрашивались;
* запрос выполняется;
* данные получены;
* произошла ошибка;
* получен пустой список.

Это состояние помогает интерфейсу показать индикатор загрузки, результат или сообщение об ошибке.

### Состояние полученных данных

Содержит информацию, которую frontend получил с сервера.

Например:

* список услуг;
* профиль пользователя;
* записи на приём;
* товары каталога;
* комментарии;
* результаты поиска.

Такие данные могут временно находиться в приложении, пока пользователь работает со страницей.

### Общее состояние приложения

Некоторые данные нужны сразу нескольким частям интерфейса.

Например:

* текущий пользователь;
* содержимое корзины;
* выбранный язык;
* тема оформления;
* количество новых уведомлений.

Если корзина отображается и в каталоге, и в шапке сайта, обе части интерфейса должны получать согласованные данные о её состоянии.

## Локальное и общее состояние

Состояние может принадлежать одному компоненту или использоваться во всём приложении.

### Локальное состояние

Локальное состояние нужно только отдельному элементу или небольшой части страницы.

Например:

* открыто ли конкретное модальное окно;
* какой пункт раскрыт в списке;
* что введено в поле поиска;
* наведён ли курсор на элемент.

Если эти данные не нужны другим частям сайта, нет смысла хранить их на уровне всего приложения.

### Общее состояние

Общее состояние используется несколькими страницами или компонентами.

Например:

* авторизованный пользователь;
* корзина;
* избранные товары;
* настройки интерфейса;
* текущая организация или филиал.

Такое состояние должно быть доступно всем элементам, которым оно необходимо.

Важно не превращать любые данные в общее состояние. Чем больше частей приложения могут его менять, тем сложнее понимать, откуда появилось конкретное значение.

## Временное состояние и постоянные данные

Состояние приложения не всегда сохраняется надолго.

Например, пользователь открыл меню. Если он обновит страницу, меню, скорее всего, снова закроется. Это временное состояние интерфейса.

Но данные профиля пользователя должны сохраниться даже после закрытия браузера. Обычно они находятся на сервере или в базе данных.

Важно различать:

* временное состояние frontend;
* данные, сохранённые в браузере;
* данные, сохранённые на сервере;
* данные, сохранённые в базе данных.

Например, содержимое корзины может храниться по-разному:

* только в состоянии страницы до её обновления;
* в браузере пользователя;
* в аккаунте на сервере;
* одновременно в браузере и на сервере.

Выбор зависит от требований проекта.

Само наличие данных в состоянии ещё не означает, что они сохранены постоянно.

## Что происходит после обновления страницы

Обычное состояние JavaScript часто исчезает после обновления или закрытия страницы.

Приложение запускается заново и создаёт начальное состояние.

Затем оно может восстановить нужные данные:

* получить профиль пользователя с сервера;
* прочитать настройки из браузера;
* загрузить корзину через API;
* определить текущую страницу по адресу;
* получить параметры поиска из URL.

Поэтому фраза «сайт помнит данные» может означать разные процессы.

Иногда приложение действительно хранит значение во время текущей работы. В других случаях оно заново получает или восстанавливает его после запуска.

При постановке задачи ИИ важно уточнять, должно ли значение сохраняться после:

* перехода на другую страницу;
* обновления страницы;
* закрытия вкладки;
* закрытия браузера;
* входа с другого устройства.

Это разные требования и разные способы реализации.

## Состояние и сервер

Frontend и сервер могут одновременно работать с похожими данными, но выполняют разные задачи.

Например, пользователь изменяет имя в профиле.

Сначала новое имя появляется в состоянии формы. Затем frontend отправляет его серверу. Сервер проверяет данные и сохраняет изменение в базе. После успешного ответа frontend обновляет отображаемый профиль.

Получается такая последовательность:

1. Пользователь вводит новое значение.
2. Значение находится в состоянии формы.
3. Frontend отправляет его через API.
4. Backend проверяет и сохраняет данные.
5. Сервер возвращает ответ.
6. Frontend обновляет состояние интерфейса.

Frontend не должен считать важные данные сохранёнными только потому, что они появились на экране. Подтверждением обычно служит успешный ответ сервера.

## Состояние и URL

Некоторую информацию можно хранить в адресе страницы.

Например, URL может содержать:

* поисковый запрос;
* выбранную категорию;
* номер страницы;
* параметры сортировки;
* идентификатор товара;
* активный раздел.

Это полезно, потому что пользователь может:

* скопировать ссылку;
* открыть её в другой вкладке;
* вернуться к странице позже;
* поделиться выбранными фильтрами;
* использовать кнопки браузера «Назад» и «Вперёд».

Если фильтры существуют только во временном состоянии, после обновления страницы они могут исчезнуть. Если они записаны в URL, приложение может восстановить их.

Не каждое состояние нужно переносить в адрес. Например, нет смысла указывать в URL, открыта ли небольшая подсказка. Но важные параметры страницы часто полезно отражать в ссылке.

## Состояние загрузки и ошибок

Одна из частых задач — правильно описать состояние запроса к серверу.

Недостаточно хранить только полученные данные. Приложению также нужно понимать, что происходит вокруг запроса.

Например, страница со списком услуг может находиться в четырёх состояниях:

### Загрузка

Запрос отправлен, но ответ ещё не получен. Интерфейс показывает индикатор загрузки.

### Успех

Сервер вернул услуги. Интерфейс показывает карточки.

### Пустой результат

Запрос выполнен успешно, но услуг нет. Интерфейс показывает отдельное сообщение.

### Ошибка

Запрос завершился неудачно. Интерфейс объясняет проблему и может предложить повторную загрузку.

Пустой результат и ошибка — не одно и то же. Пустой список может быть правильным ответом сервера, а ошибка означает, что запрос не удалось нормально обработать.

## Состояние и компоненты

Современные интерфейсы часто состоят из компонентов: кнопок, форм, карточек, списков и других самостоятельных элементов.

Состояние определяет, какие данные компонент получает и что показывает.

Например, карточка записи на приём может получить:

* дату;
* время;
* имя специалиста;
* статус записи.

При этом сама карточка может хранить локальное состояние:

* раскрыты ли подробности;
* открыто ли окно отмены;
* выполняется ли запрос на изменение записи.

При проектировании важно понимать:

* где должно находиться состояние;
* какой компонент его изменяет;
* каким компонентам оно нужно;
* что произойдёт после изменения.

Если одни и те же данные независимо хранятся в нескольких местах, они могут перестать совпадать. Например, в шапке сайта будет указано три товара, а на странице корзины — четыре.

Поэтому для одних данных желательно иметь один понятный источник актуального состояния.

## Почему состояние может становиться сложным

В небольшом интерфейсе состояние выглядит просто. Есть кнопка, значение и изменение на экране.

В крупном приложении разные состояния начинают влиять друг на друга.

Например:

* выбранный город меняет список филиалов;
* выбранный филиал меняет доступных специалистов;
* выбранный специалист меняет свободные даты;
* выбранная дата меняет доступное время;
* изменение услуги может сбросить предыдущий выбор.

Если эти зависимости не продуманы, пользователь может увидеть несовместимые данные.

Например, выбрать время, которое не относится к новому специалисту.

Поэтому важно не только перечислить значения состояния, но и определить правила:

* что от чего зависит;
* какое действие меняет данные;
* что нужно сбросить после изменения;
* какие значения должны сохраняться;
* когда нужно обращаться к серверу.

## Почему это важно при работе с ИИ-ассистентом

ИИ-ассистент может создать интерфейс, но без понятного описания состояния он будет вынужден самостоятельно придумывать поведение сайта.

Фраза:

> Сделай форму записи.

не объясняет:

* сколько у формы шагов;
* какие значения нужно помнить;
* можно ли вернуться назад;
* сохраняются ли введённые данные;
* когда выполняется запрос;
* что показывать во время загрузки;
* что происходит после ошибки;
* нужно ли очищать форму после успеха.

Более точная постановка задачи может выглядеть так:

> Создай форму записи из трёх шагов. Сохраняй выбранную услугу, специалиста, дату и время при переходе между шагами. При смене специалиста очищай ранее выбранные дату и время. Во время отправки блокируй кнопку, а после успешного ответа показывай подтверждение.

Такой промт описывает не только внешний вид, но и состояние приложения.

При проверке ответа ИИ полезно спросить:

* какие состояния используются;
* где они хранятся;
* какие действия их изменяют;
* какие компоненты получают эти данные;
* что сохраняется после обновления страницы;
* что загружается с сервера;
* как обрабатываются загрузка и ошибки;
* нет ли нескольких копий одних данных;
* какие значения сбрасываются при изменении связанных полей.

Понимание состояния помогает тебе обсуждать с ИИ поведение приложения, а не только расположение элементов на странице.

## Простая аналогия

Состояние приложения можно сравнить с рабочим столом.

На столе находятся документы, открытые папки, заметки и предметы, которые нужны тебе прямо сейчас. Их расположение показывает текущую ситуацию: над чем ты работаешь, что уже выбрано и какой шаг будет следующим.

Архив или шкаф в этой аналогии похож на базу данных. Информация там хранится постоянно, даже когда работа закончена.

Рабочий стол — это текущее состояние. Архив — постоянное хранение.

После завершения работы часть информации со стола можно убрать, а важные документы сохранить в архиве.

## Частые заблуждения новичков

### Состояние — это любые данные сайта

Не совсем. Состояние описывает текущую ситуацию, которая влияет на работу или отображение приложения. Не все файлы и данные проекта являются состоянием.

### Состояние всегда хранится в базе данных

Нет. Многие состояния существуют только во frontend: открытое меню, активная вкладка или текст незавершённого поиска.

### Если данные показаны на экране, они уже сохранены

Нет. Изменение может существовать только в состоянии интерфейса. Для постоянного сохранения часто требуется успешный запрос к серверу.

### После обновления страницы состояние обязательно сохранится

Нет. Обычное временное состояние часто исчезает. Для восстановления данных нужен отдельный механизм.

### Все состояния нужно хранить в одном месте

Нет. Локальные данные удобнее держать рядом с компонентом, которому они принадлежат. Общими должны становиться только данные, действительно нужные нескольким частям приложения.

### Одни данные можно без проблем хранить в нескольких местах

Это может привести к расхождениям. Лучше понимать, где находится основной источник актуальных данных.

### Состояние и интерфейс — одно и то же

Нет. Состояние содержит текущие значения, а интерфейс показывает их пользователю.

### ИИ сам определит правильную логику состояния

ИИ может предложить вариант, но не знает всех требований проекта. Человек должен описать, что приложение помнит, когда изменяет и когда сбрасывает.

## Что важно запомнить

* Состояние описывает текущую ситуацию в приложении.
* Оно может содержать введённые данные, выбранные элементы, результаты API, загрузку и ошибки.
* Изменение состояния обычно приводит к обновлению интерфейса.
* Состояние может быть локальным или общим.
* Временное состояние не обязательно сохраняется после обновления страницы.
* Состояние frontend и постоянные данные на сервере — разные вещи.
* Для важных данных нужно определить основной источник актуальной информации.
* При работе с API следует учитывать загрузку, успешный ответ, пустой результат и ошибку.
* ИИ-ассистенту нужно объяснять, какие состояния существуют и какие действия их изменяют.
* Человек определяет логику приложения и проверяет, правильно ли ИИ организовал работу с состоянием.

## Промты для самостоятельного уточнения

* «Объясни мне простыми словами разницу между состоянием интерфейса, данными сервера и базой данных.»

* «Покажи на примере формы записи, какие состояния в ней могут существовать и как они связаны между собой.»

* «Помоги определить локальное и общее состояние для этого интерфейса: [опиши страницы и компоненты].»

* «Объясни, какие данные исчезнут после обновления страницы и какими способами приложение может их восстановить.»

* «Разбери этот компонент и объясни, какое состояние он хранит, кто его изменяет и какие части интерфейса от него зависят: [вставь код].»

* «Предложи структуру состояний загрузки, успеха, пустого результата и ошибки для запроса к API.»

* «Проверь архитектуру состояния в моём приложении и найди данные, которые хранятся в нескольких местах и могут перестать совпадать.»

* «Помоги составить промт для ИИ-ассистента, чтобы он реализовал состояние многошаговой формы и не терял введённые данные между шагами.»
