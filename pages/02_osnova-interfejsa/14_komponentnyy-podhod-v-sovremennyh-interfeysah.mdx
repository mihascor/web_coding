# Компонентный подход: из чего собираются современные интерфейсы

## Коротко

Современный веб-интерфейс обычно собирается из отдельных частей — компонентов.

Компонентом может быть:

* кнопка;
* поле формы;
* карточка услуги;
* меню;
* уведомление;
* модальное окно;
* шапка сайта;
* форма записи.

Каждый компонент решает определённую задачу. Он может иметь собственную структуру, оформление, поведение и состояния.

Один компонент можно использовать несколько раз с разными данными. Например, вместо создания каждой карточки специалиста с нуля разработчик создаёт общий компонент и передаёт ему имя, фотографию, специализацию и стоимость.

Компонентный подход помогает не дублировать код, сохранять единый интерфейс и изменять отдельные части проекта без полной переработки страницы.

## Что такое компонент

Компонент — это самостоятельная часть интерфейса с понятным назначением.

Например, карточка услуги может содержать:

```text
Карточка услуги
├── Название
├── Описание
├── Цена
└── Кнопка записи
```

На странице может находиться несколько таких карточек. Их структура и оформление одинаковы, но содержание различается.

Компонент можно представить как шаблон, по которому создаются похожие элементы.

Например, один компонент карточки используется для разных услуг:

* консультации;
* процедуры;
* программы ухода.

Так разработчику не приходится копировать одинаковую разметку и логику.

## Для чего нужен компонентный подход

### Повторное использование

Один компонент можно применять в разных частях проекта.

Общая кнопка может использоваться:

* в форме;
* в карточке;
* в модальном окне;
* в личном кабинете.

Если её оформление изменится, достаточно обновить один компонент.

### Единообразие

Повторно используемые компоненты помогают сохранять одинаковые:

* цвета;
* размеры;
* отступы;
* состояния;
* правила поведения.

Пользователь быстрее понимает интерфейс, если похожие элементы выглядят и работают одинаково.

### Разделение сложной страницы

Большую страницу проще воспринимать как набор блоков:

```text
Страница услуги
├── Шапка
├── Главный экран
├── Описание
├── Карточки специалистов
├── Форма записи
└── Подвал
```

Каждую часть можно разрабатывать и проверять отдельно.

### Упрощение изменений

Если ошибка находится в форме записи, не нужно переписывать всю страницу. Можно изменить компонент формы или конкретное поле.

## Из чего состоит компонент

В зависимости от технологии компонент может быть устроен по-разному, но обычно включает несколько частей.

### Структура

Определяет, какие HTML-элементы находятся внутри компонента.

Например, уведомление может содержать:

* заголовок;
* текст;
* иконку;
* кнопку закрытия.

### Оформление

CSS определяет:

* цвет;
* размер;
* отступы;
* расположение;
* адаптивное поведение;
* внешний вид разных состояний.

### Поведение

JavaScript определяет, как компонент реагирует на действия пользователя.

Например, модальное окно может открываться, закрываться и реагировать на клавишу Escape.

### Входные данные

Компонент может получать информацию извне.

Карточке услуги можно передать:

* название;
* описание;
* цену;
* изображение;
* доступность записи.

В React такие входные параметры часто называют **props**.

### Состояние

Состояние показывает, что происходит с компонентом сейчас.

Например:

* меню открыто или закрыто;
* форма отправляется;
* кнопка отключена;
* данные загружаются;
* возникла ошибка.

## Компонент и его экземпляр

Компонент — это общее описание.

Экземпляр компонента — его конкретное использование.

Например, `Карточка специалиста` является компонентом. Карточка конкретного врача с его именем, фотографией и специализацией — экземпляром этого компонента.

Один компонент может создать множество экземпляров с разными данными.

Это помогает отделить структуру интерфейса от его содержимого.

## Композиция компонентов

Сложные компоненты собираются из более простых.

Например:

```text
Форма записи
├── Поле имени
├── Поле телефона
├── Выбор услуги
├── Выбор даты
├── Сообщение об ошибке
└── Кнопка отправки
```

Такое соединение называется **композицией**.

Каждая часть решает собственную задачу, а вместе они создают полноценную форму.

При этом не каждый HTML-элемент нужно превращать в отдельный компонент. Обычный абзац или декоративная оболочка могут оставаться частью более крупного блока.

Компонент имеет смысл выделять, если он:

* повторяется;
* имеет отдельное назначение;
* содержит собственное поведение;
* имеет несколько состояний;
* может изменяться независимо;
* используется в разных местах.

## Ответственность компонента

У компонента должна быть понятная задача.

Например:

* кнопка запускает действие;
* поле принимает значение;
* карточка показывает одну сущность;
* список отображает набор карточек;
* форма собирает и отправляет данные.

Если один компонент одновременно загружает данные, управляет меню, рассчитывает стоимость и отправляет форму, его становится трудно понимать.

Такой компонент стоит разделить на несколько смысловых частей.

Но чрезмерное дробление тоже мешает. Если каждый заголовок и небольшая оболочка находятся в отдельном файле, структура проекта становится сложнее без пользы.

Главный ориентир:

> Компонент должен иметь самостоятельный смысл и понятную причину для изменения.

## Родительские и дочерние компоненты

Компоненты могут находиться друг внутри друга.

Например:

```text
Каталог услуг
└── Карточка услуги
    └── Кнопка записи
```

Каталог является родительским компонентом для карточки.

Карточка является дочерним компонентом каталога и родительским для кнопки.

Родитель может передавать дочернему компоненту:

* данные;
* настройки;
* текущее состояние;
* функцию для обработки действия.

Например, каталог передаёт карточке данные услуги, а карточка сообщает каталогу, что пользователь нажал кнопку записи.

Так строится обмен информацией между частями интерфейса.

## Данные и состояние

В компонентном интерфейсе важно понимать, где хранятся данные.

Например, страница записи может хранить:

* выбранную услугу;
* выбранного специалиста;
* дату;
* контактные данные.

Дочерние компоненты получают нужную часть информации и сообщают о действиях пользователя.

Упрощённо поток выглядит так:

```text
Данные страницы
↓
Компонент
↓
Действие пользователя
↓
Обновление данных
↓
Новый вид интерфейса
```

Если одинаковое состояние хранится в нескольких местах, данные могут расходиться.

Например, одна часть страницы считает услугу выбранной, а другая показывает старое значение.

Поэтому должно быть понятно:

* где находится главный источник данных;
* кто может их изменять;
* какие компоненты только отображают результат.

## Компоненты и основные технологии

Компонентный подход не заменяет HTML, CSS и JavaScript.

**HTML** создаёт структуру и смысл элементов.

**CSS** отвечает за оформление компонента и его состояния.

**JavaScript** управляет поведением и данными.

Например, компонент кнопки должен использовать настоящий HTML-элемент `<button>`, а не обычный блок с обработчиком клика. Это сохраняет стандартное управление с клавиатуры и понятный смысл элемента.

Компонент должен учитывать:

* обычное состояние;
* наведение;
* фокус;
* отключение;
* загрузку;
* ошибку;
* мобильный экран.

Если базовый компонент реализован правильно, он улучшает сразу все места, где используется.

## Компоненты и фреймворки

Компонентный подход часто используется в:

* React;
* Vue;
* Angular;
* Svelte;
* Next.js.

Но компоненты существуют не только во фреймворках.

Даже в обычном HTML-проекте можно:

* создавать повторяемые шаблоны;
* использовать общие CSS-классы;
* подключать одинаковые блоки на разных страницах;
* создавать элементы через JavaScript-функции.

Фреймворки помогают организовать компоненты, данные и обновление интерфейса, но сама идея компонентного подхода от них не зависит.

При работе с ИИ нужно указывать технологию проекта. Компонент для обычного JavaScript будет отличаться от компонента React или Vue.

## Почему это важно при работе с ИИ

Без компонентного подхода легко попросить ИИ переписать всю страницу ради небольшого изменения.

Слабый запрос:

> Измени кнопки на странице.

Более точный запрос:

> В проекте уже есть компонент `Button`. Добавь ему второстепенный вариант оформления. Не меняй существующие варианты и предусмотрите состояния hover, focus, disabled и loading.

Так ИИ понимает:

* какой компонент нужно изменить;
* что должно сохраниться;
* какие состояния учесть;
* что не нужно переписывать всю страницу.

При создании компонента полезно указывать:

* назначение;
* входные данные;
* состояния;
* действия пользователя;
* результат действия;
* используемую технологию;
* ограничения проекта.

Пример:

> Создай компонент карточки услуги для React. Он получает название, описание, цену и идентификатор. При нажатии на кнопку передаёт идентификатор через `onSelect`. Используй существующий компонент кнопки и не загружай данные внутри карточки.

## Как проверять компонент, созданный ИИ

Проверь несколько вопросов.

### Понятно ли назначение

Можно ли одним предложением объяснить, за что отвечает компонент?

### Не существует ли уже похожего компонента

ИИ может создать новую кнопку или карточку вместо использования готовой.

### Понятны ли входные данные

Названия параметров должны соответствовать их назначению.

### Учтены ли состояния

Нужно проверить загрузку, ошибку, отсутствие данных и отключённое состояние.

### Сохранился ли правильный HTML

Кнопка должна оставаться кнопкой, ссылка — ссылкой, поле — полем формы.

### Не выполняет ли компонент слишком много задач

Если в нём много несвязанных функций, его стоит разделить.

### Не разделён ли интерфейс слишком мелко

Отдельные компоненты должны иметь самостоятельный смысл.

### Соответствуют ли стили проекту

Не нужно добавлять новую систему оформления, если проект уже использует другую.

### Работает ли компонент в разных условиях

Нужно проверить мобильный экран, длинный текст, отсутствующие данные и повторные действия.

## Простая аналогия

Компонентный подход можно сравнить со сборкой кухни из готовых модулей.

Есть шкафы, полки, ящики, ручки и столешницы. Каждый модуль имеет своё назначение и может использоваться в разных комбинациях.

Если нужно заменить ручки, не требуется переделывать всю кухню.

Так же и в интерфейсе: изменение общего компонента кнопки может обновить кнопки на нескольких страницах.

Но один модуль не должен одновременно выполнять роль шкафа, стола и холодильника. Компоненту тоже нужна понятная ответственность.

## Частые заблуждения

### «Компонент — это любой HTML-тег»

Компонент обычно объединяет структуру, оформление, данные и поведение для решения определённой задачи.

### «Компоненты существуют только в React»

Компонентный подход используется и в других фреймворках, шаблонах и обычных веб-проектах.

### «Чем меньше компонент, тем лучше»

Слишком мелкое разделение усложняет структуру без заметной пользы.

### «Один универсальный компонент лучше нескольких»

Слишком универсальный компонент получает множество условий и становится трудным для поддержки.

### «Компонент должен сам загружать данные»

Часто компоненту лучше передавать уже подготовленные данные.

### «ИИ сам определит правильные границы»

Без контекста ИИ может создать один огромный компонент или слишком много мелких.

## Что важно запомнить

* Современный интерфейс собирается из компонентов.
* Компонент имеет понятное назначение и может использоваться повторно.
* Компоненты объединяют структуру, оформление, поведение и данные.
* Конкретное использование компонента называется его экземпляром.
* Сложные блоки собираются из простых с помощью композиции.
* Не каждый HTML-элемент нужно превращать в компонент.
* У каждого компонента должна быть понятная ответственность.
* Родительские компоненты передают данные дочерним и получают информацию об их действиях.
* Должно быть понятно, где находится главный источник состояния.
* Компоненты не отменяют HTML, CSS, JavaScript и доступность.
* Компонентный подход используется как с фреймворками, так и без них.
* При работе с ИИ нужно указывать назначение, данные, состояния, действия и ограничения.
* Человек определяет структуру проекта, а ИИ помогает реализовать и улучшить компоненты.

## Промты для самостоятельного уточнения

* «Объясни компонентный подход на примере страницы онлайн-записи.»

* «Покажи дерево компонентов этой страницы и ответственность каждой части.»

* «Объясни разницу между компонентом, его экземпляром, входными данными и состоянием.»

* «Определи, какие части страницы стоит вынести в компоненты.»

* «Проверь, не выполняет ли этот компонент слишком много задач.»

* «Раздели большой компонент по ответственности без лишнего дробления.»

* «Проверь, можно ли использовать существующие компоненты вместо создания новых.»

* «Помоги составить описание компонента: назначение, данные, состояния, действия и ограничения.»
