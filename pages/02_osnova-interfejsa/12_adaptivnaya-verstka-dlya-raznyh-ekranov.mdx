# Адаптивная верстка: как сайт подстраивается под разные экраны

## Коротко

Пользователи открывают сайты на смартфонах, планшетах, ноутбуках и больших мониторах. Размер экрана, способ управления и доступное пространство при этом отличаются.

Адаптивная верстка позволяет странице подстраиваться под эти условия. Элементы могут менять размер и расположение, колонки — перестраиваться, меню — становиться компактнее, а изображения — уменьшаться без выхода за границы экрана.

Адаптивность не означает создание отдельного сайта для каждого устройства. Обычно используется одна HTML-структура, а CSS определяет, как она должна выглядеть при разной ширине экрана.

Понимание адаптивной верстки помогает ставить ИИ-ассистенту точные задачи. Вместо просьбы «сделай мобильную версию» ты можешь описать, как должны перестраиваться блоки, какие элементы остаются важными и что нужно проверить на небольшом экране.

## Что это такое

**Адаптивная верстка** — это подход к созданию интерфейса, при котором страница удобно отображается на экранах разного размера.

Один и тот же сайт может выглядеть по-разному:

* на большом мониторе меню располагается в одну строку;
* на планшете часть элементов становится компактнее;
* на смартфоне меню скрывается за кнопкой;
* сетка из четырёх карточек превращается в две или одну колонку;
* крупные отступы уменьшаются;
* изображения подстраиваются под ширину контейнера.

При этом содержание и основные функции сайта обычно остаются теми же.

Адаптивность строится не вокруг конкретных моделей устройств. Невозможно заранее создать отдельное оформление для каждого смартфона, планшета или ноутбука.

Вместо этого интерфейс проектируют так, чтобы он гибко реагировал на доступное пространство.

Основная идея выглядит так:

> Элементы должны занимать столько места, сколько им доступно, а при нехватке пространства — перестраиваться по заранее заданным правилам.

## Для чего это используется

Адаптивная верстка нужна, чтобы человек мог пользоваться сайтом независимо от устройства.

На небольшом экране пользователь должен иметь возможность:

* прочитать текст без постоянного увеличения;
* нажать кнопку пальцем;
* заполнить форму;
* открыть меню;
* увидеть важную информацию;
* прокрутить страницу только по вертикали;
* не сталкиваться с блоками, выходящими за пределы экрана.

На большом экране интерфейс тоже должен оставаться удобным.

Например, текстовый абзац не стоит растягивать на всю ширину широкого монитора. Слишком длинные строки труднее читать. Поэтому содержимое часто помещают в контейнер с ограниченной максимальной шириной.

Адаптивная верстка помогает:

* сделать сайт удобным на мобильных устройствах;
* сохранить читаемость текста;
* избежать горизонтальной прокрутки;
* правильно расположить кнопки и формы;
* использовать доступное пространство;
* поддерживать единый сайт вместо нескольких отдельных версий;
* улучшить доступность интерфейса;
* упростить развитие проекта.

Адаптивность важна не только для публичных сайтов. Она нужна личным кабинетам, административным панелям, интернет-магазинам, системам записи и внутренним сервисам.

Даже если сотрудники обычно работают за компьютером, им может понадобиться открыть систему с планшета или телефона.

## Как это работает в общей картине

Адаптивная верстка в основном создаётся с помощью CSS.

HTML определяет структуру страницы:

* шапку;
* меню;
* основной контент;
* карточки;
* формы;
* подвал.

CSS определяет:

* размеры элементов;
* количество колонок;
* отступы;
* расположение;
* размер текста;
* поведение изображений;
* изменения при разной ширине экрана.

JavaScript может использоваться для дополнительного поведения. Например, он открывает и закрывает мобильное меню.

Общая схема выглядит так:

1. Браузер получает HTML.
2. Создаёт DOM-структуру страницы.
3. Загружает CSS.
4. Определяет доступную ширину окна.
5. Применяет подходящие правила оформления.
6. Перестраивает элементы.
7. При изменении размера окна пересчитывает оформление.

Например, CSS может содержать общее правило:

> Карточки располагаются в три колонки.

И дополнительное условие:

> Если экран стал уже определённого значения, карточки располагаются в одну колонку.

Браузер сам выбирает подходящий набор правил.

## Область просмотра страницы

Чтобы понять адаптивность, важно знать понятие **viewport**.

Viewport — это видимая область страницы внутри браузера.

На компьютере viewport обычно близок к размеру окна браузера. На смартфоне он соответствует области экрана, доступной для отображения сайта.

В HTML-документ обычно добавляется специальная настройка:

```html
<meta name="viewport" content="width=device-width, initial-scale=1">
```

Она сообщает мобильному браузеру, что ширину страницы нужно сопоставить с шириной устройства.

Без этой настройки мобильный браузер может представить страницу как уменьшенную версию широкого сайта. В результате текст и элементы станут слишком мелкими.

Для начинающего важно запомнить: корректная настройка viewport — одна из базовых частей мобильной верстки.

При создании полноценного HTML-документа ИИ обычно должен добавить её автоматически. Но это всё равно стоит проверять.

## Гибкая ширина элементов

В неадаптивной верстке элементам могут задаваться жёсткие размеры.

Например, блок шириной 1200 пикселей будет нормально выглядеть на широком экране, но не поместится на смартфоне шириной около нескольких сотен пикселей.

В адаптивном интерфейсе часто используются гибкие размеры.

Например:

```css
.container {
  width: 100%;
  max-width: 1200px;
}
```

Такой контейнер:

* занимает доступную ширину;
* уменьшается на небольшом экране;
* перестаёт расширяться после достижения максимального размера.

Это распространённый принцип адаптивной верстки:

> Использовать доступное пространство, но ограничивать чрезмерное растягивание.

Для внутренних отступов обычно добавляют пространство по краям, чтобы содержимое не прижималось к границе экрана.

Например, на широком экране контейнер может иметь большие боковые отступы, а на смартфоне — небольшие.

## Относительные и фиксированные размеры

В CSS размеры можно задавать разными единицами.

### Пиксели

Пиксели подходят для многих конкретных значений:

* толщины границы;
* небольших отступов;
* размера иконки;
* минимальной высоты кнопки.

Но если всё построить только на жёстких размерах, интерфейс может плохо подстраиваться под пространство.

### Проценты

Проценты задают размер относительно родительского элемента.

Например, ширина `100%` обычно означает, что элемент занимает всю доступную ширину контейнера.

### Относительные единицы

В CSS также используются единицы, связанные с размером текста или окна браузера.

Они помогают создавать более гибкие интерфейсы, но применять их нужно осознанно.

Необязательно сразу запоминать все единицы. Важно понимать разницу:

* фиксированный размер задаёт конкретное значение;
* относительный размер зависит от другого элемента, текста или экрана;
* ограничение максимального и минимального размера защищает интерфейс от чрезмерного сжатия или растягивания.

При работе с ИИ можно попросить:

> Используй гибкие размеры контейнеров и изображений. Не задавай основным блокам фиксированную ширину, которая может вызвать горизонтальную прокрутку.

## Медиазапросы

**Медиазапросы** — это CSS-условия, которые применяют стили только при определённых характеристиках экрана.

Чаще всего условием становится ширина области просмотра.

Пример:

```css
.cards {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
}

@media (max-width: 768px) {
  .cards {
    grid-template-columns: 1fr;
  }
}
```

Этот код условно означает:

* на широком экране карточки располагаются в три колонки;
* на экране не шире 768 пикселей — в одну колонку.

Значение, при котором меняется оформление, называют **точкой перелома**, или **breakpoint**.

Медиазапросы могут учитывать не только ширину, но для базовой адаптивной верстки именно она используется чаще всего.

Важно не воспринимать точку перелома как название конкретного устройства.

Например, значение 768 пикселей не означает строго «планшет». Это просто ширина, при которой текущая композиция больше не помещается удобно.

Хороший принцип:

> Точки перелома выбирают по поведению контента, а не только по популярным размерам устройств.

Если меню перестало помещаться в строку при ширине 850 пикселей, его можно перестроить именно в этот момент, даже если в готовом шаблоне использовалось другое значение.

## Подход mobile first

**Mobile first** означает, что сначала описывается базовое оформление для небольшого экрана, а затем добавляются изменения для более широких экранов.

Например, базовый стиль создаёт одну колонку:

```css
.cards {
  display: grid;
  grid-template-columns: 1fr;
}
```

Затем для широкого экрана добавляется несколько колонок:

```css
@media (min-width: 900px) {
  .cards {
    grid-template-columns: repeat(3, 1fr);
  }
}
```

Логика mobile first выглядит так:

1. Сначала определить самое важное содержание.
2. Разместить его в ограниченном пространстве.
3. Сделать основные действия удобными.
4. Затем расширять композицию по мере появления свободного места.

Этот подход помогает не воспринимать мобильную версию как уменьшенную копию компьютерной страницы.

На смартфоне часто требуется по-другому организовать интерфейс:

* поставить блоки друг под другом;
* сделать кнопки шире;
* сократить декоративные элементы;
* изменить расположение меню;
* уменьшить отступы;
* сохранить удобный размер текста.

Mobile first — не обязательное правило для каждого проекта, но это распространённый и полезный способ мыслить об адаптивности.

## Desktop first

Существует и обратный подход — **desktop first**.

В этом случае сначала создаётся оформление для широкого экрана, а затем через медиазапросы оно упрощается для более узких экранов.

Оба подхода могут привести к рабочему результату.

Главное — соблюдать последовательность и не превращать CSS в набор конфликтующих исправлений.

Проблема возникает, когда стили добавляются хаотично:

* сначала для компьютера;
* затем несколько исправлений для телефона;
* потом отдельное исправление для планшета;
* затем новое правило, отменяющее старое.

В таком коде становится трудно понять, какое правило отвечает за итоговый вид элемента.

Если проект использует определённый подход, ИИ стоит сообщить об этом:

> Проект построен по mobile first. Сохрани этот подход и добавляй медиазапросы через `min-width`.

## Гибкие сетки

Сетка определяет, как блоки располагаются относительно друг друга.

Для адаптивных сеток часто используются:

* Flexbox;
* CSS Grid.

### Flexbox

Flexbox подходит для расположения элементов в строку или столбец.

Он может:

* переносить элементы на новую строку;
* распределять свободное пространство;
* выравнивать элементы;
* менять направление расположения.

Например, кнопки могут находиться рядом на широком экране и располагаться друг под другом на смартфоне.

### CSS Grid

Grid удобен для сеток из строк и колонок.

Он подходит для:

* карточек услуг;
* каталогов;
* галерей;
* блоков преимуществ;
* общей композиции страницы.

Сетка может автоматически подбирать количество колонок в зависимости от доступной ширины.

Условно можно задать правило:

> Создавай столько колонок, сколько помещается, но не делай карточку уже определённого размера.

Такой подход иногда позволяет сократить количество медиазапросов.

Важно не выбирать Flexbox или Grid только потому, что один инструмент кажется современнее. У каждого из них есть подходящие задачи.

## Перестройка, а не уменьшение

Одна из распространённых ошибок — попытка просто уменьшить компьютерную страницу.

Например:

* уменьшить весь текст;
* уменьшить кнопки;
* уменьшить изображения;
* сократить отступы;
* оставить пять колонок.

Формально всё может поместиться, но пользоваться такой страницей будет неудобно.

На небольшом экране интерфейс часто нужно именно перестроить.

Например:

* боковая панель перемещается под основное содержание;
* горизонтальное меню становится раскрывающимся;
* таблица получает другой способ представления;
* карточки располагаются в одну колонку;
* второстепенная информация раскрывается по нажатию;
* основные действия перемещаются ближе к пользователю.

Адаптивность — это не уменьшение масштаба, а изменение композиции под доступное пространство.

## Контент определяет верстку

Адаптивность зависит не только от ширины экрана, но и от содержимого.

Например, в карточке может быть:

* короткое название «Уход»;
* длинное название «Комплексная программа восстановления кожи».

Если интерфейс проверялся только с коротким текстом, длинный заголовок может:

* вытолкнуть кнопку;
* увеличить высоту карточки;
* выйти за границы;
* наложиться на соседний элемент.

Проблемы также возникают из-за:

* длинных имён;
* больших чисел;
* нескольких строк адреса;
* перевода сайта на другой язык;
* сообщений об ошибках;
* отсутствующего изображения;
* большого количества пунктов меню.

Поэтому адаптивную страницу нужно проверять не только на разных экранах, но и с разным содержимым.

В промте для ИИ можно указать:

> Учти длинные заголовки и описания. Текст не должен выходить за границы карточки, а кнопки должны оставаться доступными.

## Адаптивные изображения

Изображение может быть шире мобильного экрана и создать горизонтальную прокрутку.

Базовое правило часто выглядит так:

```css
img {
  max-width: 100%;
  height: auto;
}
```

Оно означает:

* изображение не становится шире доступного контейнера;
* высота изменяется пропорционально;
* изображение не искажается.

Но одной гибкой ширины не всегда достаточно.

На разных экранах могут понадобиться:

* изображения разного размера;
* разные варианты кадрирования;
* изменение пропорций;
* отдельные изображения для мобильной и широкой версии;
* оптимизация объёма загружаемого файла.

Например, широкая фотография хорошо выглядит в компьютерном баннере, но на смартфоне главный объект может оказаться слишком мелким. Иногда мобильной версии нужен другой кадр.

Важно различать:

* изменение размера изображения;
* обрезку изображения;
* замену изображения на другой вариант.

ИИ стоит сообщить, какой результат требуется:

> На широком экране изображение должно заполнять левую половину блока. На мобильном экране перемести его над текстом и сохрани главный объект в центре кадра.

## Адаптивная типографика

Размер текста тоже должен учитывать экран.

Слишком крупный заголовок может занять почти всю ширину смартфона. Слишком мелкий текст будет трудно читать.

При этом мобильная адаптация не означает, что весь текст нужно делать минимальным.

В адаптивной типографике учитываются:

* размер шрифта;
* высота строки;
* длина строки;
* расстояние между заголовком и текстом;
* количество строк в заголовке;
* визуальная иерархия.

На большом экране текстовый контейнер часто ограничивают по ширине, чтобы строки не становились чрезмерно длинными.

Заголовки могут плавно изменять размер между минимальным и максимальным значением.

Для этого в современном CSS существуют гибкие функции, но на базовом уровне достаточно понимать принцип:

> Размер текста должен сохранять читаемость и иерархию на любом экране.

Не стоит автоматически уменьшать все заголовки в два раза. Нужно проверять, как они выглядят вместе с реальным содержимым.

## Навигация на разных экранах

Главное меню — один из наиболее заметных адаптивных элементов.

На широком экране пункты меню обычно располагаются горизонтально.

На небольшом экране возможны разные решения:

* раскрывающееся меню;
* боковая панель;
* компактный список;
* нижняя мобильная навигация;
* горизонтальная прокрутка для отдельных категорий.

Выбор зависит от количества пунктов и назначения сайта.

Важно, чтобы мобильное меню:

* имело понятную кнопку открытия;
* закрывалось повторным нажатием;
* закрывалось по клавише Escape;
* не уходило за границы экрана;
* позволяло прокручивать длинный список;
* сохраняло доступность с клавиатуры;
* показывало своё открытое или закрытое состояние.

CSS отвечает за расположение и внешний вид меню. JavaScript обычно управляет его открытием и закрытием.

При работе с ИИ полезно описывать оба состояния:

> На ширине до 800 пикселей скрывай горизонтальное меню и показывай кнопку открытия. При открытии меню должно появляться под шапкой, не перекрывать кнопку закрытия и поддерживать управление клавиатурой.

## Кнопки и управление пальцем

На компьютере пользователь управляет точным курсором. На смартфоне он нажимает пальцем.

Поэтому интерактивные элементы должны иметь достаточно большую область нажатия.

Это относится к:

* кнопкам;
* ссылкам;
* иконкам;
* переключателям;
* пунктам меню;
* элементам выбора;
* кнопкам закрытия.

Маленькая иконка без внутренних отступов может быть неудобной, даже если визуально выглядит аккуратно.

Также не стоит располагать несколько маленьких кнопок слишком близко друг к другу.

Для мобильного интерфейса важно проверять не только вид, но и реальное взаимодействие:

* легко ли нажать нужный элемент;
* не происходит ли случайное соседнее нажатие;
* виден ли результат действия;
* не перекрывает ли экранная клавиатура поле или кнопку.

## Формы на мобильных экранах

Формы требуют отдельной проверки.

На смартфоне важно, чтобы:

* поля помещались по ширине;
* текст внутри полей был читаемым;
* подписи не исчезали;
* сообщения об ошибках не ломали сетку;
* кнопка отправки оставалась доступной;
* поля шли в понятном порядке;
* подходящая клавиатура открывалась для телефона, email или числа.

Форма из двух или трёх колонок на компьютере обычно превращается в одну колонку на смартфоне.

Нельзя полагаться только на текст внутри поля как на подпись. После ввода этот текст исчезает, и пользователь может забыть назначение поля.

Также нужно проверить страницу при открытой экранной клавиатуре. Она уменьшает доступную высоту экрана и может перекрывать часть интерфейса.

## Таблицы и сложные данные

Большие таблицы трудно разместить на мобильном экране.

Простое уменьшение текста обычно не решает проблему.

Возможные подходы:

* горизонтальная прокрутка внутри контейнера;
* скрытие второстепенных колонок;
* перестройка строк в карточки;
* раскрытие дополнительных данных;
* отдельное компактное представление.

Выбор зависит от того, какие данные важны пользователю.

Например, в таблице записей на смартфоне можно сначала показывать:

* дату;
* время;
* услугу;
* статус.

А дополнительные сведения открывать внутри записи.

Важно не скрывать данные автоматически только потому, что они не помещаются. Сначала нужно определить их значение для пользователя.

При постановке задачи ИИ можно написать:

> Сделай таблицу записей удобной на мобильном экране. Не уменьшай текст до нечитаемого размера. Сохрани дату, время и статус видимыми, а дополнительные сведения перенеси в раскрывающийся блок.

## Порядок элементов

На широком экране два блока могут находиться рядом:

* текст слева;
* изображение справа.

На смартфоне они располагаются друг под другом.

При этом нужно решить, какой блок будет первым.

Визуальный порядок можно изменить с помощью CSS, но с этим следует быть осторожным. Порядок элементов в HTML влияет на:

* чтение страницы программами экранного доступа;
* перемещение с клавиатуры;
* смысловую последовательность документа;
* отображение страницы без CSS.

Поэтому лучше сначала продумать логичный порядок в HTML, а затем оформлять его для широкого экрана.

Например, если заголовок и описание должны быть прочитаны раньше изображения, разумно расположить их раньше и в исходной структуре.

## Что скрывать на мобильной версии

На небольшом экране иногда скрывают второстепенные элементы:

* декоративные изображения;
* длинные дополнительные описания;
* некоторые подписи;
* вспомогательные ссылки.

Но скрытие не должно быть автоматическим решением любой проблемы.

Важно задать вопросы:

* Нужна ли эта информация мобильному пользователю?
* Можно ли перестроить блок вместо скрытия?
* Не исчезает ли важное действие?
* Доступен ли другой способ получить сведения?
* Не загружается ли тяжёлый файл, который всё равно скрыт?

Если CSS просто скрывает изображение, браузер в некоторых случаях всё равно может его загрузить. Поэтому визуальное скрытие не всегда решает вопрос производительности.

Нельзя скрывать обязательные условия, цену, предупреждение или важную функцию только ради компактности.

## Адаптивность и доступность

Адаптивный сайт должен учитывать не только размер устройства, но и возможности пользователя.

Человек может:

* увеличить масштаб браузера;
* увеличить системный размер текста;
* использовать клавиатуру;
* пользоваться программой экранного доступа;
* отключить анимации;
* держать устройство вертикально или горизонтально.

Интерфейс должен по возможности сохранять работоспособность в этих условиях.

Например:

* текст не должен обрезаться при увеличении;
* кнопки не должны накладываться друг на друга;
* контент должен оставаться доступным при масштабе;
* фокус не должен уходить за пределы видимой области;
* горизонтальная прокрутка не должна появляться у всей страницы без необходимости.

Адаптивность — это не проверка нескольких снимков экрана. Это способность интерфейса работать при изменении условий.

## Адаптивность и JavaScript

Большая часть адаптивного оформления выполняется с помощью CSS.

Не стоит использовать JavaScript для каждого изменения ширины или положения, если CSS может решить задачу самостоятельно.

Например, CSS подходит для:

* изменения количества колонок;
* переноса элементов;
* изменения отступов;
* скрытия декоративного блока;
* уменьшения заголовка;
* перестройки сетки.

JavaScript может понадобиться, если изменяется поведение:

* открывается мобильное меню;
* активируется другой вид сложного компонента;
* изменяется логика интерактивного элемента;
* нужно учитывать состояние приложения.

При этом код не должен определять тип устройства только по названию браузера или модели телефона. Надёжнее ориентироваться на реальные возможности и доступное пространство.

Хороший вопрос к ИИ:

> Можно ли реализовать эту адаптацию только с помощью CSS? Добавляй JavaScript только для поведения, которое CSS не может обеспечить.

## Отзывчивый и адаптивный дизайн

В русскоязычной практике слова «адаптивный» и «отзывчивый» часто используют как близкие понятия.

Иногда их различают:

* **отзывчивый дизайн** плавно подстраивается под любую доступную ширину;
* **адаптивный дизайн** переключается между несколькими заранее подготовленными вариантами.

На практике современные интерфейсы часто совмещают оба подхода:

* контейнеры имеют гибкую ширину;
* сетки плавно изменяются;
* в определённых точках композиция перестраивается.

Для начинающего важнее не спорить о терминах, а понимать ожидаемое поведение страницы.

Когда ты ставишь задачу ИИ, лучше описывать конкретный результат:

* сколько колонок должно быть;
* когда меню перестраивается;
* как ведёт себя изображение;
* какие отступы меняются;
* какие элементы остаются видимыми.

## Точки перелома

Точки перелома часто обозначают примерными категориями:

* мобильный экран;
* планшет;
* небольшой ноутбук;
* широкий экран.

Но использовать только стандартный набор значений не всегда правильно.

Точку перелома стоит добавлять там, где текущая верстка начинает работать плохо.

Признаки того, что нужна перестройка:

* меню переносится на вторую строку;
* карточки становятся слишком узкими;
* заголовок занимает неудобное количество строк;
* кнопки сталкиваются;
* форма выглядит слишком тесно;
* боковая панель оставляет мало места основному содержанию.

Слишком большое количество точек перелома тоже может усложнить CSS.

Лучше стремиться к гибкой базовой верстке и добавлять условия только там, где действительно меняется композиция.

## Как проверять адаптивность

Адаптивную страницу нельзя оценивать только по одному скриншоту.

Её нужно проверять на разных ширинах и в разных состояниях.

### Изменяй ширину постепенно

Не ограничивайся тремя готовыми размерами.

Плавно уменьшай окно браузера и смотри, в какой момент:

* появляются наложения;
* текст выходит за границы;
* меню перестаёт помещаться;
* карточки становятся слишком узкими;
* появляется горизонтальная прокрутка.

### Проверяй вертикальную и горизонтальную ориентацию

Телефон или планшет можно повернуть. Интерфейс должен корректно реагировать на изменение пропорций экрана.

### Используй реальные устройства

Режим мобильного экрана в инструментах разработчика удобен, но не полностью заменяет проверку на настоящем телефоне.

На реальном устройстве можно заметить:

* неудобные области нажатия;
* особенности экранной клавиатуры;
* проблемы с прокруткой;
* поведение браузерных панелей;
* слишком мелкие элементы.

### Проверяй разный контент

Используй:

* короткие и длинные заголовки;
* заполненную и пустую карточку;
* длинное сообщение об ошибке;
* большое количество пунктов меню;
* отсутствующее изображение;
* длинное имя пользователя.

### Проверяй интерактивные состояния

Открой:

* мобильное меню;
* модальное окно;
* выпадающий список;
* ошибки формы;
* уведомления;
* загрузку;
* пустое состояние.

Страница может выглядеть правильно в обычном состоянии и ломаться после взаимодействия.

## Инструменты браузера

Современные браузеры позволяют имитировать разные размеры экрана через инструменты разработчика.

Обычно можно:

* выбрать готовый размер устройства;
* задать произвольную ширину и высоту;
* повернуть экран;
* изменить масштаб;
* проверить сенсорное взаимодействие;
* посмотреть CSS, применённый на текущей ширине.

При этом важно не воспринимать список устройств как полный набор тестов.

Лучше использовать два способа:

1. Проверить несколько распространённых размеров.
2. Плавно менять ширину между ними.

Так можно обнаружить проблемы, которые возникают не на стандартном смартфоне, а, например, между планшетной и компьютерной версией.

## Почему это важно при работе с ИИ-ассистентом

Фраза:

> Сделай сайт адаптивным.

слишком общая.

ИИ может самостоятельно выбрать точки перелома, скрыть элементы или перестроить блоки способом, который не соответствует твоей задаче.

Лучше описать поведение основных частей страницы.

Например:

> Сделай страницу адаптивной по принципу mobile first. На мобильном экране все блоки должны идти в одну колонку. Начиная с 768 пикселей, карточки услуг должны располагаться в две колонки, а с 1100 пикселей — в три. Шапка должна превращаться в мобильное меню, изображения не должны выходить за границы контейнеров. Не скрывай цены и кнопки записи.

В таком промте указаны:

* общий подход;
* поведение сетки;
* точки перелома;
* поведение меню;
* требования к изображениям;
* важные элементы, которые нельзя скрывать.

Также полезно сообщить ИИ:

* используется ли готовый макет;
* какие размеры уже поддерживаются;
* какой CSS-подход принят в проекте;
* можно ли менять HTML;
* есть ли существующие медиазапросы;
* какие компоненты ломаются;
* на какой ширине возникает проблема;
* должен ли код быть mobile first;
* используются ли Flexbox, Grid или CSS-фреймворк.

## Как формулировать задачу по адаптивности

Задачу удобно описывать по компонентам.

### Шапка

* Как располагаются логотип и меню?
* Когда появляется кнопка мобильного меню?
* Где находится основная кнопка действия?

### Главный экран

* Что идёт первым: текст или изображение?
* Как меняется размер заголовка?
* Остаются ли кнопки рядом или становятся вертикальными?

### Карточки

* Сколько колонок используется?
* Какой должна быть минимальная ширина карточки?
* Должны ли карточки иметь одинаковую высоту?

### Формы

* Сколько колонок у полей?
* Растягивается ли кнопка на всю ширину?
* Где показываются ошибки?

### Таблицы

* Нужна ли внутренняя прокрутка?
* Какие данные остаются видимыми?
* Нужно ли перестраивать строки в карточки?

### Подвал

* Сохраняются ли колонки?
* В каком порядке располагаются разделы?
* Как отображаются длинные ссылки?

Так ИИ получает не абстрактную просьбу, а набор проверяемых правил.

## Как просить ИИ исправить конкретную проблему

При адаптивных ошибках полезно передать:

* ширину, на которой возникает проблема;
* скриншот или описание;
* HTML проблемного блока;
* связанный CSS;
* ожидаемое поведение;
* ограничения на изменения.

Например:

> При ширине от 720 до 850 пикселей пункты меню переносятся на вторую строку и увеличивают высоту шапки. Сделай так, чтобы в этом диапазоне уже использовалось мобильное меню. Не меняй структуру основного контента и существующие названия классов.

Или:

> На ширине 390 пикселей карточка выходит за пределы экрана. Найди элемент с фиксированной шириной, убери причину горизонтальной прокрутки и объясни внесённые изменения.

Чем точнее описана проблема, тем меньше вероятность, что ИИ перепишет всю страницу без необходимости.

## Как проверять адаптивный код, созданный ИИ

### Проверь отсутствие горизонтальной прокрутки

У всей страницы не должно появляться горизонтальное перемещение без обоснованной причины.

Исключением может быть отдельный контейнер таблицы или галереи, где прокрутка задумана специально.

### Проверь основные точки перелома

Посмотри интерфейс немного ниже и немного выше каждого значения. Иногда проблема появляется прямо на границе медиазапроса.

### Проверь промежуточные размеры

Страница может хорошо выглядеть на 375 и 1024 пикселях, но ломаться на 700.

### Проверь порядок содержимого

После перестройки блоки должны оставаться логичными.

### Проверь кликабельные элементы

Кнопки и ссылки должны быть удобны для управления пальцем.

### Проверь формы

Поля, подписи и ошибки не должны выходить за границы.

### Проверь изображения

Они не должны растягиваться непропорционально или создавать прокрутку.

### Проверь длинный текст

Заголовки, адреса и сообщения должны переноситься.

### Проверь открытые состояния

Меню, модальные окна и списки должны помещаться на экране.

### Проверь, что важные элементы не исчезли

ИИ мог скрыть сложный блок вместо его адаптации.

### Проверь масштабирование текста

Увеличенный текст не должен полностью разрушать интерфейс.

### Проверь существующий CSS

Новое правило не должно конфликтовать с другими медиазапросами проекта.

## Простая аналогия

Адаптивную страницу можно сравнить с модульной мебелью.

В большой комнате стол, шкафы и кресла могут стоять рядом и занимать много пространства.

В небольшой комнате те же функции сохраняются, но расположение меняется:

* стол становится компактнее;
* секции шкафа располагаются вертикально;
* часть мебели трансформируется;
* проходы остаются свободными;
* важные предметы остаются доступными.

Если просто уменьшить всю мебель в несколько раз, пользоваться ею станет неудобно.

Так же и с сайтом: мобильная версия — не уменьшенная фотография компьютерной страницы, а продуманная перестройка тех же элементов под другое пространство.

## Частые заблуждения новичков

### «Адаптивность — это отдельная мобильная страница»

Обычно используется одна страница, которая меняет оформление в зависимости от доступной ширины.

### «Достаточно уменьшить все элементы»

На небольшом экране часто требуется перестройка расположения, а не простое уменьшение.

### «Нужно поддерживать только несколько моделей телефонов»

Размеров экранов очень много. Верстка должна гибко работать между выбранными точками перелома.

### «Медиазапрос для 768 пикселей всегда означает планшет»

Точка перелома описывает ширину, а не конкретное устройство. Её лучше выбирать по поведению контента.

### «Для каждого размера нужен отдельный медиазапрос»

Хорошая гибкая сетка может работать в широком диапазоне без дополнительных условий.

### «Мобильная версия должна содержать меньше функций»

Некоторые второстепенные элементы можно упростить, но основные возможности должны сохраняться.

### «Если блок не помещается, его можно скрыть»

Скрытие может удалить важную информацию. Сначала нужно рассмотреть перенос, перестройку или раскрывающийся формат.

### «Адаптивность отвечает только за ширину»

Нужно учитывать текст, изображения, управление пальцем, ориентацию, экранную клавиатуру, масштаб и интерактивные состояния.

### «Достаточно проверить страницу в режиме одного смартфона»

Проблемы часто появляются на промежуточных ширинах или реальных устройствах.

### «JavaScript должен определять мобильное устройство»

Большинство изменений оформления надёжнее выполнять через CSS и доступное пространство.

### «Если на смартфоне нет горизонтальной прокрутки, всё готово»

Страница может помещаться, но иметь слишком мелкий текст, неудобные кнопки и нарушенный порядок элементов.

### «ИИ сам правильно решит, что скрыть»

ИИ не знает приоритеты бизнеса и потребности пользователя, пока ты их не опишешь.

### «Можно добавлять новые медиазапросы при каждой проблеме»

Большое количество локальных исправлений часто скрывает ошибку базовой структуры или фиксированных размеров.

## Что важно запомнить

* Адаптивная верстка помогает странице работать на экранах разного размера.
* Обычно используется одна HTML-структура и несколько правил отображения.
* CSS отвечает за основную перестройку интерфейса.
* Viewport определяет видимую область страницы в браузере.
* Гибкие размеры позволяют элементам использовать доступное пространство.
* Медиазапросы применяют дополнительные CSS-правила при определённых условиях.
* Точки перелома стоит выбирать по поведению контента, а не только по названиям устройств.
* Mobile first начинается с небольшой ширины и постепенно расширяет интерфейс.
* Flexbox и Grid помогают создавать гибкие сетки.
* На смартфоне страницу часто нужно перестраивать, а не просто уменьшать.
* Текст, изображения, формы, меню и таблицы требуют отдельной проверки.
* Основные функции не должны исчезать на небольшом экране.
* Адаптивность нужно проверять на промежуточных ширинах и в разных состояниях.
* JavaScript стоит использовать для поведения, а не для изменений, которые можно выполнить через CSS.
* При работе с ИИ нужно описывать поведение каждого важного блока и ограничения задачи.
* Человек определяет приоритеты интерфейса, а ИИ помогает реализовать адаптивные правила.

## Промты для самостоятельного уточнения

* «Объясни мне простыми словами, как адаптивная верстка используется в реальном веб-проекте.»

* «Покажи простой пример, как сетка из трёх карточек превращается в одну колонку на мобильном экране. Не перегружай пример кодом.»

* «Объясни, что такое viewport, медиазапросы и точки перелома и как они связаны между собой.»

* «Расскажи, чем подход mobile first отличается от desktop first и какой вариант лучше подходит моему проекту.»

* «Объясни, как Flexbox и CSS Grid помогают создавать адаптивные интерфейсы.»

* «Проверь мой HTML и CSS и найди причины горизонтальной прокрутки на мобильном экране.»

* «Помоги адаптировать эту страницу для смартфона, планшета и широкого экрана. Не скрывай важные функции и не меняй JavaScript без необходимости.»

* «Проверь адаптивность формы: размеры полей, расположение подписей, сообщения об ошибках и удобство кнопки отправки.»

* «Составь список ширин и состояний, на которых нужно проверить адаптивность этой страницы.»

* «Помоги составить точный промт для ИИ, чтобы адаптировать существующую страницу без полной переработки HTML и CSS.»
