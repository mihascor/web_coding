# API: как разные части приложения общаются друг с другом

## Коротко

Современное веб-приложение обычно состоит из нескольких частей. Интерфейс работает в браузере, сервер обрабатывает запросы, база данных хранит информацию, а внешние сервисы могут принимать платежи, отправлять сообщения или предоставлять карты.

Чтобы эти части могли обмениваться данными и командами, используется API.

Понимание API поможет тебе точнее объяснять ИИ-ассистенту, откуда приложение должно получать данные, куда их отправлять и что должно происходить после запроса.

## Что такое API

API — это заранее определённый способ взаимодействия между программами или отдельными частями приложения.

Расшифровка API — Application Programming Interface, то есть программный интерфейс приложения.

Простыми словами, API устанавливает правила общения:

* куда отправлять запрос;
* какие данные передавать;
* в каком формате их передавать;
* какие действия доступны;
* какой ответ можно получить;
* какие ошибки могут возникнуть.

Например, frontend интернет-магазина может обратиться к API сервера и попросить:

* вернуть список товаров;
* показать один товар;
* добавить товар в корзину;
* создать заказ;
* обновить профиль пользователя.

Frontend не обязан знать, как сервер ищет товары в базе данных или рассчитывает заказ. Ему достаточно знать правила обращения к API.

## Для чего используется API

API связывает разные части веб-приложения.

С его помощью можно:

* получать данные с сервера;
* отправлять заполненные формы;
* создавать и изменять записи;
* удалять данные;
* авторизовывать пользователей;
* загружать файлы;
* подключать внешние сервисы;
* передавать данные между разными приложениями.

Например, сайт онлайн-записи может использовать собственный API для получения списка специалистов и внешний API для отправки SMS-уведомлений клиентам.

API позволяет каждой части системы выполнять свою задачу, не раскрывая другим частям все внутренние детали своей работы.

## Запрос и ответ

Общение через API обычно строится по модели «запрос — ответ».

### Запрос

Одна часть приложения обращается к другой и сообщает, что она хочет получить или изменить.

Запрос может содержать:

* адрес нужной функции;
* тип действия;
* передаваемые данные;
* информацию о пользователе;
* дополнительные параметры.

Например, браузер может отправить серверу запрос на получение списка доступных записей на определённую дату.

### Ответ

Сервер обрабатывает запрос и возвращает результат.

В ответе могут находиться:

* нужные данные;
* подтверждение успешного действия;
* номер созданного заказа;
* сообщение об ошибке;
* информация о том, почему действие невозможно.

Frontend получает ответ и решает, что показать пользователю.

## Что такое endpoint

Endpoint — это конкретная точка API, к которой можно обратиться для выполнения определённого действия.

У одного API может быть несколько таких точек.

Например:

* получение списка товаров;
* получение одного товара;
* создание заказа;
* обновление профиля;
* удаление комментария.

Каждый endpoint имеет своё назначение и ожидает определённые данные.

Можно представить API как набор доступных дверей. За каждой дверью находится отдельная функция приложения. Чтобы получить нужный результат, необходимо выбрать правильную дверь и передать подходящую информацию.

## Какие действия выполняются через API

В API часто используются четыре базовых действия с данными:

* получить данные;
* создать новые данные;
* изменить существующие данные;
* удалить данные.

Например, приложение для управления задачами может через API:

* получить список задач;
* создать новую задачу;
* изменить её название или статус;
* удалить задачу.

Позже эти действия будут подробнее рассмотрены в статье о CRUD.

Кроме работы с данными, API может запускать процессы: отправлять письмо, рассчитывать стоимость доставки, проводить оплату или создавать документ.

## Как API работает в общей картине

Рассмотрим пример загрузки списка товаров.

1. Пользователь открывает страницу каталога.
2. JavaScript в браузере отправляет запрос к API.
3. Сервер получает запрос.
4. Сервер проверяет параметры и права доступа.
5. Сервер запрашивает товары из базы данных.
6. API возвращает результат браузеру.
7. Интерфейс показывает товары пользователю.

При создании заказа движение данных происходит в обратную сторону:

1. Пользователь заполняет форму.
2. Frontend проверяет введённые данные.
3. Данные отправляются через API.
4. Backend повторно проверяет информацию.
5. Заказ сохраняется в базе данных.
6. API возвращает подтверждение.
7. Интерфейс показывает результат пользователю.

Таким образом, API выступает связующим слоем между интерфейсом и серверной логикой.

## API и внешние сервисы

API используется не только внутри одного проекта.

Веб-приложение может подключаться к сторонним системам:

* платёжным сервисам;
* CRM;
* картам;
* службам доставки;
* сервисам электронной почты;
* SMS-провайдерам;
* системам аналитики;
* социальным сетям.

Например, при оплате заказа сайт передаёт платёжному сервису сумму и информацию о заказе. Сервис обрабатывает платёж и возвращает результат через API.

У каждого внешнего API есть собственные правила, ограничения и документация. Перед подключением важно понять, какие данные сервис принимает, что возвращает и как защищается доступ.

## API и формат данных

Чтобы разные системы правильно понимали друг друга, они должны использовать согласованный формат данных.

Один из самых распространённых форматов — JSON. Он позволяет передавать структурированную информацию: пользователей, товары, заказы, настройки и другие объекты.

Например, сервер может вернуть информацию о товаре, включающую:

* идентификатор;
* название;
* цену;
* описание;
* наличие;
* адрес изображения.

Frontend получает эту структуру и размещает значения в нужных частях страницы.

Формат JSON будет подробнее рассмотрен в следующей статье.

## API и безопасность

Не каждый запрос должен быть доступен любому пользователю.

API может проверять:

* вошёл ли пользователь в аккаунт;
* имеет ли он право выполнить действие;
* может ли он видеть запрашиваемые данные;
* допустимы ли переданные значения;
* не превышено ли количество запросов;
* откуда поступил запрос.

Например, обычный клиент может просматривать свои заказы, но не должен получать заказы других пользователей. Администратор может изменять каталог, а посетитель сайта — только просматривать его.

Наличие кнопки или скрытого раздела в интерфейсе не является защитой. Ограничения доступа должны проверяться сервером при каждом важном запросе к API.

## Что такое документация API

Документация API описывает правила его использования.

Обычно в ней указывается:

* какие endpoints доступны;
* какие действия они выполняют;
* какие данные нужно передавать;
* какие поля обязательны;
* что возвращается в ответе;
* как выполняется авторизация;
* какие ошибки могут возникнуть.

Документация помогает разработчику или ИИ-ассистенту понять, как правильно подключиться к API.

При работе с внешним сервисом полезно предоставить ИИ актуальную документацию или нужный её фрагмент. Иначе ассистент может использовать устаревшие или выдуманные названия функций и параметров.

## Как API связано с другими частями веб-приложения

API объединяет основные уровни проекта.

**Frontend** отправляет запросы и показывает полученные данные.

**Формы** собирают информацию для отправки.

**Валидация** проверяет данные до и после передачи.

**Backend** получает запросы и выполняет серверную логику.

**База данных** хранит информацию, которую сервер получает или изменяет.

**Внешние сервисы** предоставляют дополнительные функции через собственные API.

API не заменяет frontend, backend или базу данных. Оно определяет правила, по которым эти части взаимодействуют.

## Почему это важно при работе с ИИ-ассистентом

Задача «Подключи страницу к API» слишком общая.

ИИ-ассистенту нужно сообщить:

* к какому API обращаться;
* какой endpoint использовать;
* какие данные отправлять;
* какой ответ ожидается;
* требуется ли авторизация;
* что показывать во время загрузки;
* что делать при успешном ответе;
* как обрабатывать ошибки.

Более точный запрос может выглядеть так:

> Подключи страницу каталога к API товаров. При открытии страницы нужно получить список товаров с сервера и показать название, цену и изображение. Во время запроса покажи индикатор загрузки, а при ошибке — понятное сообщение и кнопку повторной попытки.

Так ИИ понимает не только источник данных, но и весь пользовательский сценарий.

Перед постановкой задачи полезно определить:

* кто отправляет запрос;
* кто его принимает;
* какое действие выполняется;
* какие данные передаются;
* какой результат считается успешным;
* какие ошибки нужно предусмотреть.

## Простая аналогия

API можно сравнить с официантом в ресторане.

Посетитель не идёт на кухню и не объясняет повару, как готовить блюдо. Он выбирает позицию из меню и передаёт заказ официанту.

Официант сообщает заказ кухне, а затем приносит результат или объясняет, почему блюдо сейчас недоступно.

В этой аналогии frontend — это посетитель, backend — кухня, а API — согласованный способ передачи заказа и получения ответа.

## Частые заблуждения новичков

### API — это отдельная программа

API не обязательно является самостоятельным приложением. Часто это часть backend, которая принимает запросы от frontend или других систем.

### API хранит данные

Обычно данные хранит база данных. API предоставляет правила для их получения и изменения.

### Любой API доступен бесплатно и без ограничений

Внешние API могут требовать регистрацию, оплату, ключ доступа или соблюдать лимиты запросов.

### Frontend может напрямую работать с базой данных

Обычно браузер обращается к серверному API, а уже backend взаимодействует с базой данных. Это помогает контролировать безопасность и бизнес-правила.

### Успешный запрос всегда возвращает данные

Иногда API возвращает только подтверждение выполненного действия.

### Достаточно обработать только успешный ответ

Запрос может завершиться ошибкой, потерей соединения, отказом в доступе или некорректными данными. Интерфейс должен учитывать такие ситуации.

### ИИ знает правила любого API

Конкретное API может изменяться. Для точной реализации ИИ нужна актуальная документация и контекст проекта.

## Что важно запомнить

* API задаёт правила общения между программами и частями приложения.
* Взаимодействие обычно состоит из запроса и ответа.
* Endpoint — это конкретная точка API для определённого действия.
* Frontend обращается к API, а backend обрабатывает запрос и работает с базой данных.
* Через API можно получать, создавать, изменять и удалять данные.
* Внешние сервисы также подключаются к приложению через API.
* Права доступа и валидация должны проверяться на сервере.
* При работе с ИИ нужно описывать endpoint, данные, ожидаемый ответ и обработку ошибок.
* Человек определяет правила взаимодействия, а ИИ помогает реализовать их в проекте.

## Промты для самостоятельного уточнения

* «Объясни простыми словами, как frontend, API, backend и база данных взаимодействуют в одном веб-приложении.»

* «Покажи простой сценарий запроса и ответа API без сложного кода.»

* «Объясни, что такое endpoint и чем он отличается от API в целом.»

* «Помоги определить, какие endpoints нужны для приложения онлайн-записи клиентов.»

* «Расскажи, какие данные нужно предусмотреть в API для работы с пользователями, услугами и записями.»

* «Объясни, как приложение должно обрабатывать загрузку, успешный ответ и ошибку API.»

* «Проверь моё описание функции и скажи, достаточно ли точно я указал запросы к API.»

* «Составь промт для ИИ-ассистента на подключение существующей страницы к API с обработкой загрузки и ошибок.»
