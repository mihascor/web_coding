# Клиентская и серверная часть: frontend и backend

## Коротко

Почти любой современный веб-проект состоит из двух основных частей: клиентской и серверной.

Клиентская часть, или **frontend**, работает в браузере и отвечает за то, что видит пользователь. Серверная часть, или **backend**, работает на сервере и отвечает за данные, правила и защищенные операции.

Понимание этого разделения помогает точнее ставить задачи ИИ-ассистенту. Вместо общего запроса «сделай функцию записи» ты сможешь указать, что должно происходить в интерфейсе, что — на сервере, а что — в базе данных.

## Что такое frontend

Frontend — это клиентская часть веб-проекта, с которой взаимодействует пользователь.

К frontend относятся:

* страницы сайта;
* кнопки;
* формы;
* меню;
* изображения;
* таблицы;
* уведомления;
* анимации;
* личный кабинет;
* отображение полученных данных.

Frontend работает в браузере пользователя.

Когда человек нажимает кнопку, вводит текст, открывает меню или выбирает дату, именно клиентская часть реагирует на эти действия.

Основные технологии frontend:

* **HTML** задает структуру страницы;
* **CSS** отвечает за внешний вид;
* **JavaScript** добавляет поведение и интерактивность.

В современных проектах также используются frontend-фреймворки и библиотеки, например React, Vue или Angular. Они помогают создавать сложные интерфейсы из отдельных компонентов.

## Что такое backend

Backend — это серверная часть веб-проекта.

Пользователь обычно не видит ее напрямую, но именно она выполняет важную внутреннюю работу.

Backend может:

* принимать запросы от frontend;
* проверять введенные данные;
* искать информацию в базе данных;
* сохранять новые данные;
* проверять логин и пароль;
* определять права пользователя;
* рассчитывать стоимость;
* создавать заказы;
* отправлять уведомления;
* взаимодействовать с другими сервисами;
* возвращать результат в браузер.

Например, при записи на прием frontend показывает календарь и форму, а backend проверяет, свободно ли выбранное время, сохраняет запись и возвращает подтверждение.

Для backend используются разные языки и технологии, например JavaScript, TypeScript, Python, PHP, Java, C# или Ruby.

Новичку не обязательно сразу разбираться во всех языках. Важнее понимать назначение серверной части.

## Главное отличие frontend и backend

Разницу можно выразить так:

**Frontend показывает интерфейс и принимает действия пользователя.**

**Backend обрабатывает данные и выполняет правила приложения.**

| Frontend                                  | Backend                                     |
| ----------------------------------------- | ------------------------------------------- |
| Работает в браузере                       | Работает на сервере                         |
| Видим пользователю                        | Обычно скрыт от пользователя                |
| Показывает страницы и элементы интерфейса | Обрабатывает запросы и данные               |
| Реагирует на клики и ввод                 | Проверяет и выполняет операции              |
| Отправляет запросы                        | Принимает запросы                           |
| Показывает ответ                          | Формирует ответ                             |
| Не должен хранить секреты                 | Может работать с секретными ключами         |
| Отвечает за удобство интерфейса           | Отвечает за правила и безопасность операций |

Это разделение упрощенное. В современных технологиях границы иногда могут пересекаться, но для базового понимания такая модель подходит.

## Для чего нужен frontend

Frontend нужен, чтобы человек мог удобно взаимодействовать с системой.

Без клиентской части пользователь получал бы только необработанные данные или технические ответы сервера.

Frontend превращает функции системы в понятный интерфейс.

Например, вместо отправки технического запроса пользователь видит:

* кнопку «Записаться»;
* календарь;
* список специалистов;
* форму контактных данных;
* сообщение об успешной записи.

Frontend также отвечает за состояние интерфейса.

Например, он может:

* показать индикатор загрузки;
* заблокировать кнопку после отправки;
* подсветить неправильно заполненное поле;
* обновить список без перезагрузки страницы;
* показать сообщение об ошибке;
* запомнить выбранный фильтр во время работы.

## Для чего нужен backend

Backend нужен, чтобы выполнять операции, которым нельзя доверять браузеру пользователя.

Например, сервер должен проверять:

* существует ли пользователь;
* правильный ли введен пароль;
* есть ли у пользователя нужные права;
* действительно ли товар есть в наличии;
* свободно ли выбранное время;
* можно ли изменить конкретную запись;
* правильно ли рассчитана стоимость.

Даже если frontend уже проверил данные, backend должен проверить их повторно.

Пользователь может изменить код, работающий в браузере, отключить часть проверок или отправить запрос напрямую. Поэтому важные правила нельзя оставлять только на frontend.

## Как frontend и backend работают вместе

Рассмотрим простой пример: пользователь открывает список доступных врачей.

### Шаг 1. Frontend показывает страницу

Браузер отображает заголовок, фильтры и область, в которой должен появиться список врачей.

### Шаг 2. Frontend отправляет запрос

Клиентская часть обращается к backend и просит вернуть данные.

Упрощенно запрос можно представить так:

> Верни список врачей выбранного филиала.

### Шаг 3. Backend обрабатывает запрос

Сервер:

1. Получает запрос.
2. Проверяет его параметры.
3. Обращается к базе данных.
4. Находит подходящих врачей.
5. Формирует ответ.

### Шаг 4. Backend возвращает данные

Сервер отправляет frontend список врачей, например их имена, специальности и фотографии.

### Шаг 5. Frontend отображает результат

Браузер получает данные и превращает их в карточки, которые видит пользователь.

Общая схема выглядит так:

**пользователь → frontend → backend → база данных**

Затем ответ идет обратно:

**база данных → backend → frontend → пользователь**

## Пример с формой записи

Рассмотрим, как можно разделить одну функцию между частями проекта.

### Что делает frontend

* показывает форму;
* загружает список специалистов;
* показывает доступные даты;
* проверяет, заполнены ли обязательные поля;
* отправляет данные;
* показывает состояние загрузки;
* выводит подтверждение или ошибку.

### Что делает backend

* принимает данные формы;
* проверяет их повторно;
* проверяет доступность времени;
* определяет, можно ли создать запись;
* сохраняет запись в базе данных;
* создает номер записи;
* запускает отправку уведомления;
* возвращает результат frontend.

### Что делает база данных

* хранит данные пациентов;
* хранит расписание;
* хранит записи;
* связывает запись с врачом и филиалом;
* сохраняет статус записи.

Одна пользовательская функция часто затрагивает сразу несколько частей системы.

## Что такое API

Frontend и backend обычно общаются через API.

API — это набор правил и адресов, по которым одна часть программы может обращаться к другой.

Например, backend может предоставить отдельные возможности:

* получить список врачей;
* получить свободное время;
* создать запись;
* отменить запись;
* получить профиль пользователя.

Frontend отправляет запрос к нужному адресу API и получает ответ.

Можно представить API как согласованный язык общения между клиентской и серверной частями.

Frontend должен знать:

* куда отправлять запрос;
* какие данные передавать;
* в каком формате придет ответ;
* какие ошибки возможны.

Backend должен знать:

* какие запросы принимать;
* как их проверять;
* какие действия выполнять;
* какой ответ возвращать.

## Где находится база данных

База данных не является ни интерфейсом, ни браузером. Это отдельная часть системы, которая хранит информацию.

Обычно frontend не должен обращаться к базе напрямую.

Правильная схема:

**frontend → backend → база данных**

Backend выступает контролирующим слоем. Он проверяет пользователя, применяет правила и возвращает только разрешенные данные.

Например, frontend может запросить данные текущего пользователя. Backend определит, кто отправил запрос, и вернет только его профиль.

Если разрешить браузеру напрямую обращаться ко всей базе, пользователь потенциально сможет увидеть или изменить чужую информацию.

## Почему нельзя хранить секреты во frontend

Код frontend загружается в браузер пользователя. Его можно просмотреть и исследовать.

Поэтому в клиентской части нельзя безопасно хранить:

* пароли от базы данных;
* закрытые ключи платежных систем;
* секретные ключи сторонних сервисов;
* внутренние административные токены;
* правила, которые должны оставаться скрытыми.

Иногда frontend использует публичные ключи или открытые идентификаторы. Они специально предназначены для работы в браузере и не дают полного доступа к системе.

Но настоящий секрет должен оставаться на сервере.

Если ИИ предлагает добавить закрытый ключ прямо в JavaScript-код страницы, это серьезный повод проверить решение.

## Что такое fullstack

Fullstack-разработка объединяет frontend и backend.

Fullstack-разработчик может работать:

* с интерфейсом;
* с серверной логикой;
* с API;
* с базой данных;
* с публикацией проекта.

Это не означает, что frontend и backend превращаются в одну часть. Они по-прежнему выполняют разные задачи, даже если над ними работает один человек или они находятся в одном проекте.

Существуют также fullstack-фреймворки, в которых клиентская и серверная логика организованы рядом. Например, один проект может содержать страницы, серверные обработчики и подключение к базе данных.

Но даже в таком случае важно понимать, какой код выполняется в браузере, а какой — на сервере.

## Может ли сайт работать без backend

Да. Простой информационный сайт может состоять только из frontend-файлов.

Например:

* лендинг;
* портфолио;
* страница с контактами;
* документация;
* небольшой сайт-визитка.

Такой сайт может размещаться на хостинге статических файлов и не иметь собственной серверной логики.

Однако даже простой сайт может пользоваться внешними backend-сервисами. Например, форма обратной связи может отправлять данные в сторонний сервис.

## Может ли backend работать без frontend

Да.

Backend может предоставлять API для:

* мобильного приложения;
* другого сайта;
* внутренней системы;
* кассовой программы;
* CRM;
* автоматического процесса;
* интеграции с мессенджером.

Например, один backend может одновременно обслуживать сайт, мобильное приложение и административную панель.

Поэтому backend не обязательно связан только с одним видимым интерфейсом.

## Где выполняется код

При работе с ИИ полезно постоянно задавать вопрос:

> Где будет выполняться этот код?

Возможные ответы:

* в браузере;
* на сервере;
* в базе данных;
* в процессе сборки проекта;
* во внешнем сервисе.

От места выполнения зависят:

* доступные возможности;
* уровень безопасности;
* доступ к данным;
* производительность;
* способ поиска ошибок.

Например, код frontend может управлять кнопкой и формой, но не должен напрямую использовать пароль от базы данных.

Код backend может обращаться к базе, но не может напрямую изменить уже отображенный интерфейс в браузере. Он сначала должен вернуть ответ frontend.

## Почему это важно при работе с ИИ-ассистентом

ИИ-ассистенту недостаточно сказать:

> Добавь регистрацию пользователя.

Регистрация обычно состоит из нескольких задач.

### Задачи frontend

* создать форму;
* добавить поля;
* показать правила заполнения;
* отправить данные;
* показать ошибку;
* перенаправить пользователя после регистрации.

### Задачи backend

* принять данные;
* проверить формат;
* убедиться, что пользователь еще не зарегистрирован;
* безопасно обработать пароль;
* создать запись в базе;
* вернуть ответ.

### Задачи базы данных

* хранить пользователя;
* обеспечивать уникальность электронной почты;
* сохранять необходимые поля.

Более точный промт может выглядеть так:

> Добавь регистрацию пользователя. На frontend создай форму с полями имени, электронной почты и пароля. На backend добавь обработчик регистрации, повторную проверку данных, проверку уникальности электронной почты и сохранение пользователя в базе. Не храни пароль в открытом виде. Опиши, какие файлы относятся к frontend, а какие — к backend.

Такой запрос помогает ИИ правильно разделить ответственность.

## Как проверять ответ ИИ

При оценке решения полезно проверить несколько вопросов.

### Frontend

* Есть ли понятный интерфейс?
* Обрабатываются ли загрузка и ошибки?
* Отправляются ли данные в правильном формате?
* Нет ли секретных ключей?
* Понятно ли пользователю, что происходит?

### Backend

* Проверяет ли сервер входные данные?
* Проверяются ли права доступа?
* Безопасно ли выполняется операция?
* Возвращаются ли понятные ответы?
* Обрабатываются ли ошибки базы данных?

### Связь между частями

* Совпадают ли адреса запросов?
* Совпадают ли названия полей?
* Одинаково ли понимается формат данных?
* Обрабатывает ли frontend ответы backend?
* Реализована ли функция полностью, а не только визуально?

ИИ может хорошо написать одну часть и забыть другую. Например, создать форму без серверного обработчика или API без подключения к интерфейсу.

## Простая аналогия

Frontend можно сравнить с залом ресторана.

Посетитель видит меню, столы, кнопки вызова и оформление. Он взаимодействует с тем, что находится перед ним.

Backend — это кухня и внутренние рабочие процессы.

Там проверяют заказ, готовят блюда, учитывают остатки и решают, можно ли выполнить запрос.

API похоже на правила передачи заказов между залом и кухней.

База данных напоминает систему учета продуктов, клиентов и заказов.

Красивое меню не приготовит блюдо без кухни. Но кухня без понятного меню будет неудобна для посетителя. Полноценный сервис требует согласованной работы обеих частей.

## Частые заблуждения новичков

### «Frontend — это только дизайн»

Frontend включает не только внешний вид, но и поведение интерфейса: формы, запросы, состояния, проверки и реакцию на действия пользователя.

### «Backend — это база данных»

Backend и база данных связаны, но это разные части. Backend выполняет логику, а база данных хранит информацию.

### «Если форма выглядит рабочей, данные сохраняются»

Форма может быть только визуальным макетом. Нужно проверить, отправляется ли запрос и обрабатывается ли он сервером.

### «Проверки на frontend достаточно»

Клиентская проверка улучшает удобство, но не обеспечивает безопасность. Backend должен проверять данные повторно.

### «JavaScript всегда относится к frontend»

JavaScript может выполняться и в браузере, и на сервере. Важно не название языка, а место выполнения кода.

### «Frontend может безопасно хранить секретные ключи»

Код браузера доступен пользователю. Настоящие секреты должны храниться на сервере.

### «Backend всегда должен быть отдельным проектом»

Он может находиться в отдельном приложении или в одном fullstack-проекте с frontend. Важнее разделение задач, а не количество папок.

### «ИИ сам поймет, какую часть нужно изменить»

ИИ может изменить только интерфейс, хотя задача требует серверной логики. В промте лучше прямо перечислять затронутые части.

## Что важно запомнить

* Frontend — это клиентская часть, которая работает в браузере.
* Backend — это серверная часть, которая обрабатывает запросы, правила и данные.
* Frontend отвечает за интерфейс и взаимодействие с пользователем.
* Backend отвечает за серверную логику, безопасность и доступ к данным.
* База данных обычно подключается через backend.
* Frontend и backend общаются через API.
* Важные проверки должны выполняться на сервере, даже если они уже есть в браузере.
* Секретные ключи нельзя хранить во frontend.
* При постановке задачи ИИ нужно указывать, какие изменения требуются на frontend, backend и в базе данных.
* Человек определяет архитектуру и проверяет, полностью ли ИИ реализовал пользовательский сценарий.

## Промты для самостоятельного уточнения

* «Объясни простыми словами разницу между frontend, backend и базой данных на примере реального веб-проекта».
* «Раздели функцию записи на прием на задачи для frontend, backend и базы данных».
* «Покажи, как frontend и backend общаются через API, но без сложного кода».
* «Объясни, какие проверки нужно выполнять в браузере, а какие обязательно повторять на сервере».
* «Помоги определить, нужен ли моему проекту собственный backend».
* «Проверь описание моей функции и скажи, какие части должны выполняться на frontend, а какие — на backend».
* «Объясни, почему секретные ключи нельзя хранить в коде frontend».
* «Составь промт для ИИ-ассистента, чтобы он реализовал функцию сразу на frontend, backend и в базе данных».
