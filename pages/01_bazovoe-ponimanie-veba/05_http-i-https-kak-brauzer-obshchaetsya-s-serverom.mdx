# HTTP и HTTPS: как браузер общается с сервером

## Коротко

Браузеру и серверу нужны общие правила общения. Эти правила определяет протокол **HTTP**.

Когда ты открываешь страницу, отправляешь форму или загружаешь данные, браузер формирует HTTP-запрос. Сервер принимает его, выполняет нужное действие и возвращает HTTP-ответ.

**HTTPS** — это защищенная версия такого общения. Она шифрует передаваемые данные и помогает браузеру убедиться, что он подключился к настоящему серверу сайта.

Понимание HTTP и HTTPS помогает точнее ставить задачи ИИ-ассистенту, разбираться в запросах к API и находить причины ошибок между frontend и backend.

## Что такое HTTP

HTTP расшифровывается как **Hypertext Transfer Protocol** — протокол передачи гипертекста.

Протокол — это набор правил, по которым две системы обмениваются информацией.

HTTP определяет:

* как браузер формирует запрос;
* как указывается нужная страница или операция;
* как передаются дополнительные параметры;
* как сервер формирует ответ;
* как сообщается об успехе или ошибке;
* в каком формате отправляются данные.

Несмотря на слово «гипертекст» в названии, HTTP используется не только для HTML-страниц. Через него можно передавать:

* JSON-данные;
* изображения;
* видео;
* документы;
* CSS;
* JavaScript;
* содержимое форм;
* другие типы файлов.

HTTP можно представить как общий формат переписки между клиентом и сервером.

## Что такое HTTPS

HTTPS расшифровывается как **Hypertext Transfer Protocol Secure** — защищенный протокол передачи данных.

По своей логике HTTPS использует те же запросы и ответы, что и HTTP. Главное отличие заключается в защищенном соединении.

HTTPS помогает обеспечить три важных свойства:

* **шифрование** — посторонние не должны прочитать передаваемые данные;
* **целостность** — данные не должны быть незаметно изменены по пути;
* **проверку подлинности** — браузер проверяет, что подключается к нужному сайту.

Например, когда пользователь вводит пароль или данные банковской карты, HTTPS защищает информацию во время передачи между браузером и сервером.

Сегодня HTTPS нужен не только интернет-магазинам и банкам. Он является базовым требованием практически для любого современного сайта.

## Для чего нужны HTTP и HTTPS

Браузер использует HTTP или HTTPS почти при любом взаимодействии с сервером.

Например, когда пользователь:

* открывает страницу;
* входит в аккаунт;
* отправляет форму;
* загружает фотографию;
* получает список товаров;
* сохраняет настройки;
* оформляет заказ;
* нажимает кнопку удаления;
* загружает документ.

Каждое такое действие может создавать один или несколько запросов.

Frontend отправляет запрос, backend его обрабатывает, а затем возвращает ответ.

Общая схема:

**frontend → HTTP-запрос → backend**

**backend → HTTP-ответ → frontend**

Если используется HTTPS, передача между ними дополнительно защищена шифрованием.

## Из чего состоит HTTP-запрос

HTTP-запрос содержит несколько частей.

На базовом уровне важно понимать четыре элемента:

1. Адрес запроса.
2. Метод запроса.
3. Заголовки.
4. Тело запроса.

Не в каждом запросе присутствуют все эти части.

## Адрес запроса

Адрес указывает, куда именно обращается браузер.

Например:

`https://example.com/api/doctors`

Такой адрес можно условно разделить на части:

* `https` — протокол;
* `example.com` — домен;
* `/api/doctors` — путь к нужному ресурсу.

Путь помогает backend определить, какие данные или действия запрашиваются.

Например:

* `/api/doctors` — получить врачей;
* `/api/appointments` — работать с записями;
* `/api/profile` — получить профиль пользователя;
* `/api/login` — выполнить вход.

Такие адреса часто называют маршрутами или endpoints.

## Методы HTTP-запросов

Метод сообщает серверу, какое действие хочет выполнить клиент.

Наиболее распространенные методы:

### GET

Используется для получения данных.

Например:

* получить страницу;
* загрузить список врачей;
* получить данные профиля;
* показать историю заказов.

GET-запрос обычно не должен изменять данные на сервере.

### POST

Используется для отправки новых данных или запуска операции.

Например:

* зарегистрировать пользователя;
* создать запись на прием;
* отправить форму;
* оформить заказ.

### PUT

Часто используется для полной замены существующего объекта.

Например, полностью обновить данные профиля.

### PATCH

Используется для частичного изменения данных.

Например, изменить только номер телефона пользователя.

### DELETE

Используется для удаления данных.

Например:

* отменить запись;
* удалить товар;
* удалить комментарий.

Эти значения описывают намерение запроса. Конкретное поведение все равно определяется кодом backend.

Сервер технически может удалять данные через POST, но понятное использование методов делает API предсказуемее.

## Заголовки запроса

Заголовки, или headers, передают дополнительную информацию о запросе.

Они могут сообщать:

* какой тип данных отправляется;
* какой формат ответа ожидается;
* кто отправляет запрос;
* какой язык предпочитает пользователь;
* есть ли данные авторизации;
* можно ли использовать кеш;
* с какого сайта пришел запрос.

Например, заголовок может указывать, что тело запроса содержит JSON.

Другой заголовок может содержать токен авторизации, по которому сервер определит пользователя.

Заголовки не отображаются как обычное содержимое страницы, но играют важную роль в общении frontend и backend.

## Тело запроса

Тело запроса содержит данные, которые клиент отправляет серверу.

Оно часто используется с методами POST, PUT и PATCH.

Например, при создании записи на прием браузер может передать:

* идентификатор врача;
* выбранную дату;
* время;
* имя пациента;
* номер телефона;
* комментарий.

Данные могут передаваться в разных форматах. В современных API часто используется JSON.

При загрузке файлов применяется другой формат, позволяющий передавать изображения, документы и дополнительные поля формы.

## Параметры адреса

Некоторые данные могут передаваться прямо в адресе.

Например:

`/api/doctors?branch=central&speciality=cosmetology`

Часть после знака `?` называется строкой запроса, или query string.

Она часто используется для:

* фильтрации;
* поиска;
* сортировки;
* выбора страницы списка;
* передачи необязательных параметров.

Такие параметры хорошо подходят для получения данных, но не для хранения секретной информации. Адрес может сохраняться в истории браузера, логах и аналитических системах.

## Из чего состоит HTTP-ответ

Сервер возвращает HTTP-ответ.

В нем можно выделить:

1. Статус ответа.
2. Заголовки.
3. Тело ответа.

## Статус ответа

Статус показывает результат обработки запроса.

Он состоит из трехзначного числа.

Распространенные группы статусов:

* **1xx** — информационные сообщения;
* **2xx** — запрос выполнен успешно;
* **3xx** — требуется перенаправление или другое действие;
* **4xx** — проблема связана с запросом клиента;
* **5xx** — ошибка произошла на стороне сервера.

Новичку полезно знать несколько основных статусов.

### 200 OK

Запрос успешно выполнен.

Например, сервер вернул список врачей.

### 201 Created

Новый объект успешно создан.

Например, создана запись на прием.

### 204 No Content

Запрос выполнен, но серверу нечего возвращать в теле ответа.

Например, запись успешно удалена.

### 301 и 302

Сервер сообщает, что нужно перейти по другому адресу.

Это называется перенаправлением.

### 400 Bad Request

Сервер считает запрос некорректным.

Например, данные отправлены в неправильном формате.

### 401 Unauthorized

Для выполнения запроса требуется вход или действительная авторизация.

Несмотря на название, этот статус обычно означает, что пользователь не подтвержден.

### 403 Forbidden

Сервер узнал пользователя, но не разрешает ему выполнить действие.

Например, обычный сотрудник пытается открыть раздел администратора.

### 404 Not Found

Запрошенная страница, маршрут или объект не найдены.

### 422 Unprocessable Content

Запрос имеет понятный формат, но содержит недопустимые данные.

Например, указан некорректный адрес электронной почты или выбранное время уже занято.

### 500 Internal Server Error

На сервере произошла внутренняя ошибка.

Это общий статус, который не раскрывает точную причину проблемы.

### 502 и 503

Обычно указывают на недоступность сервера или связанных с ним систем.

Например, backend временно не работает или промежуточный сервер не может получить от него ответ.

## Тело ответа

Тело содержит сам результат.

Это может быть:

* HTML-страница;
* JSON с данными;
* изображение;
* файл;
* подтверждение операции;
* описание ошибки.

Например, запрос списка врачей может вернуть массив объектов с именами и специальностями.

Если произошла ошибка, backend может вернуть пояснение:

> Выбранное время уже занято.

Frontend получает ответ и решает, как показать его пользователю.

## Как frontend обрабатывает ответ

После получения ответа frontend обычно проверяет его статус.

Если запрос успешен, интерфейс может:

* показать данные;
* обновить список;
* закрыть форму;
* вывести подтверждение;
* перейти на другую страницу.

Если сервер вернул ошибку, frontend может:

* подсветить поле;
* показать сообщение;
* предложить повторить запрос;
* перенаправить на страницу входа;
* скрыть запрещенную функцию;
* записать техническую информацию для диагностики.

Важно, чтобы backend возвращал понятные статусы, а frontend правильно их обрабатывал.

Если сервер сообщает об ошибке, но интерфейс всегда показывает «Все успешно», пользователь получает неверную информацию.

## Как работает HTTPS

При использовании HTTPS браузер и сервер сначала устанавливают защищенное соединение.

Для этого применяется технология TLS.

В упрощенном виде процесс выглядит так:

1. Браузер обращается к серверу.
2. Сервер показывает цифровой сертификат.
3. Браузер проверяет сертификат.
4. Стороны договариваются о ключах шифрования.
5. Дальнейшие запросы и ответы передаются в зашифрованном виде.

Браузер проверяет:

* для какого домена выдан сертификат;
* не истек ли срок его действия;
* можно ли доверять организации, которая его выдала;
* не был ли сертификат отозван;
* соответствует ли он текущему подключению.

Пользователь обычно видит результат этой проверки рядом с адресной строкой.

## Что дает шифрование

Без шифрования данные могут быть прочитаны тем, кто получил доступ к сетевому трафику.

Например, это особенно опасно в публичной сети Wi-Fi.

HTTPS защищает во время передачи:

* пароли;
* контактные данные;
* содержимое форм;
* cookies;
* токены;
* платежные данные;
* ответы сервера;
* страницы, которые открывает пользователь.

Однако HTTPS защищает только канал передачи.

Он не гарантирует, что:

* сам сайт безопасен;
* сервер не был взломан;
* данные правильно хранятся в базе;
* пользователь не отправляет информацию мошенническому сайту;
* в коде нет уязвимостей;
* владелец сайта действует добросовестно.

HTTPS — важная часть безопасности, но не вся безопасность целиком.

## HTTP и HTTPS — не разные сайты

Иногда один домен доступен по двум адресам:

* `http://example.com`
* `https://example.com`

Технически это разные варианты подключения.

Обычно сервер перенаправляет пользователя с HTTP на HTTPS, чтобы все данные передавались защищенно.

В правильно настроенном проекте пользователь должен работать только через HTTPS.

## Что такое сертификат

Сертификат — это электронный документ, который связывает домен с его владельцем или сервером.

Он используется для установки доверенного HTTPS-соединения.

Сертификат содержит техническую информацию:

* домен;
* срок действия;
* данные организации, выдавшей сертификат;
* открытый ключ;
* цифровую подпись.

Сертификаты имеют ограниченный срок действия, поэтому их нужно регулярно обновлять.

Во многих современных хостингах выпуск и обновление сертификатов автоматизированы.

## Что такое API-запрос

Когда frontend запрашивает у backend не готовую страницу, а данные или выполнение операции, это часто называют API-запросом.

Например:

* получить расписание;
* создать пользователя;
* изменить запись;
* удалить товар;
* отправить сообщение.

API-запросы используют тот же HTTP или HTTPS.

Разница не в протоколе, а в назначении ответа. Вместо готовой HTML-страницы сервер часто возвращает JSON, который frontend использует для обновления интерфейса.

## Пример работы формы

Рассмотрим отправку формы записи.

### 1. Пользователь заполняет форму

Frontend получает имя, телефон, врача и выбранное время.

### 2. Frontend выполняет первичную проверку

Браузер проверяет, заполнены ли обязательные поля и похож ли номер телефона на допустимый.

### 3. Frontend создает запрос

Данные отправляются на backend методом POST через HTTPS.

### 4. Backend проверяет запрос

Сервер проверяет:

* корректность данных;
* доступность времени;
* права пользователя;
* отсутствие дублирующей записи.

### 5. Backend сохраняет данные

Если все правильно, сервер создает запись в базе данных.

### 6. Backend возвращает ответ

Например:

* статус 201 — запись создана;
* статус 422 — время уже занято;
* статус 500 — внутренняя ошибка сервера.

### 7. Frontend показывает результат

Интерфейс выводит подтверждение или понятное сообщение об ошибке.

Так HTTP связывает пользовательский интерфейс с серверной логикой.

## Почему сервер должен проверять данные

Frontend может проверить форму до отправки, но этой проверки недостаточно.

Код в браузере находится под контролем пользователя. Проверку можно обойти и отправить запрос напрямую.

Поэтому backend должен самостоятельно проверить:

* обязательные поля;
* формат данных;
* допустимые значения;
* права пользователя;
* ограничения бизнес-логики;
* наличие связанных объектов.

Например, даже если frontend не показывает занятое время, сервер все равно должен убедиться, что оно остается свободным в момент создания записи.

HTTP передает запрос, но именно backend решает, можно ли его выполнить.

## Что такое cookies и авторизация

Браузер может отправлять серверу cookies вместе с HTTP-запросом.

Cookies — это небольшие данные, которые сайт сохраняет в браузере.

Они могут использоваться для:

* поддержания авторизации;
* хранения идентификатора сессии;
* запоминания настроек;
* работы корзины;
* аналитики.

Другой распространенный вариант — передача токена в заголовке запроса.

Сервер проверяет cookie или токен и определяет:

* кто отправил запрос;
* вошел ли пользователь;
* какие действия ему разрешены.

HTTPS особенно важен для авторизованных запросов, потому что защищает идентификаторы сессии и токены во время передачи.

## Что такое CORS

Иногда frontend и backend работают на разных доменах или адресах.

Например:

* frontend: `app.example.com`;
* backend: `api.example.com`.

Браузер применяет правила CORS, которые определяют, может ли один сайт обращаться к другому источнику.

Если backend не разрешил такой запрос, браузер может его заблокировать.

Ошибка CORS часто сбивает новичков с толку, потому что backend может быть доступен, но браузер все равно не передает ответ frontend.

Важно понимать: CORS — это механизм браузера. Он не заменяет авторизацию и не является полной защитой API.

## Что такое тайм-аут

Браузер или другая программа не может ждать ответ бесконечно.

Если сервер отвечает слишком долго, запрос может завершиться по тайм-ауту.

Причиной могут быть:

* медленная база данных;
* зависшая серверная операция;
* недоступный внешний сервис;
* плохое сетевое соединение;
* перегруженный сервер.

Frontend должен уметь обрабатывать такие ситуации и сообщать пользователю, что запрос не удалось выполнить.

## Почему запрос может повторяться

Иногда один запрос отправляется несколько раз.

Причины могут быть нормальными:

* автоматическая повторная попытка;
* обновление данных;
* проверка авторизации;
* повторное открытие компонента.

Но возможны и ошибки:

* обработчик события подключен дважды;
* пользователь несколько раз нажал кнопку;
* JavaScript попал в цикл;
* интерфейс повторно запускает загрузку;
* браузер повторяет предварительный CORS-запрос.

При работе с ИИ важно сообщать не только текст ошибки, но и количество запросов, методы, адреса и статусы.

## Почему это важно при работе с ИИ-ассистентом

Многие задачи веб-разработки связаны с HTTP, даже если это не указано прямо.

Например, запрос:

> Сделай сохранение формы.

может включать:

* выбор адреса API;
* выбор HTTP-метода;
* создание тела запроса;
* настройку заголовков;
* серверную проверку;
* сохранение в базе;
* выбор статуса ответа;
* обработку результата на frontend.

Полезный промт должен описывать обе стороны общения.

Например:

> Добавь отправку формы записи через HTTPS. Frontend должен отправлять POST-запрос в формате JSON. Backend должен проверить обязательные поля, сохранить запись и вернуть статус 201. Если время занято, верни статус 422 и понятное сообщение, которое frontend покажет пользователю.

Так ИИ понимает не только внешний результат, но и правила взаимодействия.

## Как анализировать ошибку запроса

Когда запрос не работает, полезно определить:

* какой адрес вызывается;
* какой метод используется;
* какие данные отправляются;
* какие заголовки передаются;
* какой статус возвращает сервер;
* что находится в теле ответа;
* появляется ли ошибка в консоли;
* дошел ли запрос до backend;
* есть ли связанная ошибка в серверных логах.

Например, фраза:

> API не работает.

слишком общая.

Гораздо полезнее:

> Frontend отправляет POST-запрос на `/api/appointments`. Сервер возвращает статус 400 и сообщение о том, что поле `doctorId` отсутствует, хотя форма содержит выбранного врача. Помоги проверить формат тела запроса и названия полей.

Это дает ИИ конкретный материал для анализа.

## Простая аналогия

HTTP можно сравнить с оформлением заказа по стандартному бланку.

В бланке указано:

* куда направить заказ;
* какое действие требуется;
* кто отправитель;
* какие данные приложены;
* в каком виде нужен ответ.

Сервер получает бланк, обрабатывает его и возвращает квитанцию с результатом.

HTTPS напоминает передачу этого бланка в защищенном конверте. Посторонний не должен прочитать или изменить содержимое по дороге.

При этом защищенный конверт не гарантирует, что сам магазин работает честно или правильно обрабатывает заказы. Он защищает именно передачу.

## Частые заблуждения новичков

### «HTTP — это интернет»

HTTP работает через интернет, но не является самим интернетом. Это один из протоколов передачи данных.

### «HTTPS означает, что сайт полностью безопасен»

HTTPS защищает соединение, но не устраняет ошибки кода, слабые пароли и уязвимости сервера.

### «GET используется для любых запросов»

GET предназначен прежде всего для получения данных. Создание, изменение и удаление обычно оформляют другими методами.

### «Статус 200 означает, что пользователь получил правильный результат»

Он означает успешную обработку на уровне HTTP. Но данные в ответе или логика интерфейса все равно могут быть ошибочными.

### «Статус 500 объясняет причину ошибки»

500 только сообщает о внутренней проблеме сервера. Точную причину нужно искать в серверных логах.

### «Если frontend проверил форму, серверу проверять ее не нужно»

Frontend-проверку можно обойти. Важные ограничения должны проверяться на backend.

### «HTTPS скрывает адрес сайта»

Данные запроса шифруются, но часть технической информации о соединении все равно может быть видна сетевым системам.

### «Все данные нужно передавать в адресе»

Параметры адреса подходят не для всех задач. Пароли, токены и чувствительные данные не следует помещать в URL.

### «Ошибка CORS означает, что сервер сломан»

Сервер может работать, но браузер блокирует доступ из-за политики источников и отсутствующих разрешений.

### «ИИ достаточно попросить подключить API»

Для хорошего результата нужно описать метод, адрес, формат данных, авторизацию, успешный ответ и возможные ошибки.

## Что важно запомнить

* HTTP задает правила общения между клиентом и сервером.
* HTTPS — это HTTP через защищенное зашифрованное соединение.
* Браузер отправляет запрос, а сервер возвращает ответ.
* Запрос содержит адрес, метод, заголовки и иногда тело.
* Ответ содержит статус, заголовки и содержимое.
* GET обычно получает данные, POST создает или запускает операцию, PATCH изменяет, DELETE удаляет.
* Статусы 2xx означают успех, 4xx — проблему с запросом, 5xx — проблему на сервере.
* HTTPS защищает передачу, но не делает весь сайт автоматически безопасным.
* Backend должен повторно проверять данные, даже если frontend уже выполнил проверку.
* При работе с ИИ нужно описывать обе стороны запроса: что отправляет frontend и что должен вернуть backend.
* Для диагностики важны адрес, метод, данные, статус ответа и сообщение об ошибке.
