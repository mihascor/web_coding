# pnpm vs pnm

## Коротко

**npm** и **pnpm** делают похожую работу: устанавливают пакеты, читают `package.json`, запускают скрипты и управляют зависимостями.

Главное отличие:

```text
npm — стандартный и самый привычный менеджер пакетов
pnpm — более быстрый и экономный по месту менеджер пакетов
```

Оба работают с тем же npm Registry — большим хранилищем JavaScript-пакетов. npm Registry содержит более двух миллионов пакетов, по данным npm. ([npm](https://www.npmjs.com/?utm_source=chatgpt.com "npm | Home"))

---

# Главное различие простыми словами

Когда ты устанавливаешь зависимости через npm, он обычно кладёт копии пакетов в папку проекта `node_modules`.

А **pnpm** хранит пакеты в общем хранилище на компьютере и в проекты добавляет ссылки на эти пакеты. Документация pnpm описывает это как **content-addressable store**: зависимости сохраняются в общем хранилище, а одинаковые файлы не дублируются лишний раз. ([pnpm.io](https://pnpm.io/motivation?utm_source=chatgpt.com "Motivation"))

Условно:

```text
npm:
project-a/node_modules/react
project-b/node_modules/react
project-c/node_modules/react

pnpm:
общее хранилище/react
project-a → ссылка на react
project-b → ссылка на react
project-c → ссылка на react
```

Поэтому pnpm обычно:

- быстрее устанавливает зависимости;
    
- занимает меньше места на диске;
    
- строже относится к зависимостям.

# Сравнение npm и pnpm

|Критерий|npm|pnpm|
|---|---|---|
|Назначение|Установка и управление пакетами|То же самое|
|Использует `package.json`|Да|Да|
|Использует npm Registry|Да|Да|
|Папка зависимостей|`node_modules`|тоже `node_modules`, но устроена иначе|
|Lock-файл|`package-lock.json`|`pnpm-lock.yaml`|
|Скорость|Нормальная|часто быстрее|
|Место на диске|Может занимать больше|обычно экономит место|
|Строгость зависимостей|Более привычное, мягкое поведение|более строгое|
|Простота для новичка|Проще начать|чуть больше нюансов|
|Популярность|Стандарт по умолчанию с Node.js|часто используют в современных проектах и монорепозиториях|

# Отличие в командах

Команды очень похожи.

## Установка всех зависимостей

npm:

```bash
npm install
```

pnpm:

```bash
pnpm install
```

---

## Установка пакета

npm:

```bash
npm install axios
```

pnpm:

```bash
pnpm add axios
```

---

## Установка dev-зависимости

npm:

```bash
npm install vite --save-dev
```

pnpm:

```bash
pnpm add vite -D
```

---

## Запуск скрипта

npm:

```bash
npm run dev
```

pnpm:

```bash
pnpm dev
```

или так:

```bash
pnpm run dev
```

# Чем отличается `node_modules`

Это важный момент.

В npm `node_modules` обычно выглядит более “плоско”: многие зависимости могут оказаться доступными на верхнем уровне проекта.

В pnpm структура `node_modules` более строгая. pnpm делает ссылки на пакеты и старается, чтобы проект мог использовать только те зависимости, которые явно указаны в `package.json`. Официальный сайт pnpm отдельно подчёркивает скорость, экономию диска и удобство для монорепозиториев. ([pnpm.io](https://pnpm.io/?utm_source=chatgpt.com "Fast, disk space efficient package manager | pnpm"))

Практический смысл:

```js
import something from "some-package";
```

Если `some-package` не указан в `package.json`, npm иногда может позволить такому коду случайно работать из-за особенностей структуры `node_modules`.

А pnpm чаще покажет проблему: пакет не установлен явно.

Это полезно, потому что проект становится честнее и предсказуемее.

# Lock-файлы отличаются

У npm:

```text
package-lock.json
```

У pnpm:

```text
pnpm-lock.yaml
```

Lock-файл фиксирует точные версии зависимостей, чтобы проект одинаково устанавливался на разных компьютерах. Документация npm указывает, что `package-lock.json` автоматически создаётся при операциях, где npm изменяет `node_modules` или `package.json`. ([npm Documentation](https://docs.npmjs.com/cli/v8/configuring-npm/package-lock-json/?utm_source=chatgpt.com "package-lock.json"))

Важно: **не стоит одновременно использовать в одном проекте npm и pnpm без необходимости**.

Иначе могут появиться сразу разные lock-файлы:

```text
package-lock.json
pnpm-lock.yaml
```

Это может запутать проект и команду.

# Что лучше для новичка

Для начала я бы советовал так:

## Если ты только учишься

Начни с **npm**.

Причины:

- он устанавливается вместе с Node.js;
    
- чаще встречается в обучающих материалах;
    
- большинство инструкций в интернете пишут сначала под npm;
    
- проще понять базовую логику `package.json`, `node_modules`, `npm install`, `npm run dev`.
    

---

## Когда уже уверенно понимаешь npm

Можно попробовать **pnpm**.

Особенно если:

- проект большой;
    
- много зависимостей;
    
- несколько проектов на одном компьютере;
    
- используется монорепозиторий;
    
- хочется быстрее устанавливать пакеты;
    
- хочется экономить место на диске.

# Простая аналогия

Представь, что пакеты — это книги.

**npm** чаще кладёт нужные книги прямо в папку каждого проекта.

**pnpm** хранит книги в одной общей библиотеке, а в каждый проект кладёт аккуратные ссылки: “эта книга лежит вот там”.

Поэтому pnpm не копирует одно и то же много раз.

# Что важно запомнить

**npm** — стандартный менеджер пакетов, с него проще начинать.

**pnpm** — альтернатива npm, которая делает установку зависимостей быстрее и экономнее.

Оба используют `package.json`.

Главные отличия pnpm:

```text
быстрее
экономит место
строже проверяет зависимости
использует pnpm-lock.yaml вместо package-lock.json
иначе организует node_modules
```

Для начинающего веб-кодера достаточно понимать так:

```text
npm — базовый вариант
pnpm — более продвинутый и оптимизированный вариант
```

На практике, если в проекте есть `package-lock.json`, обычно используют npm.  
Если есть `pnpm-lock.yaml`, используют pnpm.
